Close the two verified gaps in Phase 1 (`01-VERIFICATION.md`) that block phase-goal
achievement. Both were reproduced live against a real Postgres and a real HTTP request chain;
this plan makes both reproductions pass and adds automated regressions so they cannot silently
re-open.

- **Gap 1 (BASE-01 / ROADMAP SC1):** `internal/jobs/repo_test.go`'s `TestJobLifecycle` and
  `TestMarkFailed` call `Repo.Create` with a `CreateParams` that omits `ClientID` (zero-value
  `uuid.Nil`). Since Plan 02 threaded `client_id` into the `jobs` INSERT behind a
  `REFERENCES clients(id)` FK, inserting `uuid.Nil` violates `jobs_client_id_fkey`
  (SQLSTATE 23503) whenever the tests actually run (i.e. `DATABASE_URL` set). The tests silently
  skipped in every prior sandbox, so "`go test ./...` passes" was never truly validated.

- **Gap 2 (RATE-03 / ROADMAP SC5, == 01-REVIEW.md CR-01):** `internal/api/routes.go` installs
  chi's `middleware.RealIP` globally before `ratelimit.ByIP`. `RealIP` is deprecated by chi
  v5.3.0 for being spoofable — it overwrites `r.RemoteAddr` from client-supplied
  `X-Forwarded-For`/`X-Real-IP`/`True-Client-IP`. `ByIP` keys on `httprate.KeyByIP` over that
  mutated address, so varying the forwarded header per request lands each request in a fresh
  bucket and fully evades the coarse pre-auth flood guard.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated client → API `/v1` (pre-auth) | Untrusted HTTP input, including fully attacker-controlled request headers, crosses here before any API key is validated. The coarse `ratelimit.ByIP` guard is the only control acting before the auth resolver's Postgres lookup. |
| API process → Postgres `jobs`/`clients` | The `jobs_client_id_fkey` foreign key is a data-integrity boundary; only real client ids may be attributed to jobs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | `internal/api/routes.go` IP middleware + `internal/ratelimit/ratelimit.go` `ByIP` key derivation | mitigate | Drop `middleware.RealIP`; use `middleware.ClientIPFromRemoteAddr` (trusts only the raw TCP peer address) and key `ByIP` on `middleware.GetClientIP`, so client-supplied `X-Forwarded-For`/`X-Real-IP`/`True-Client-IP` can no longer set the rate-limit key. Enforced by the router-level regression test in Task 2. |
| T-01-02 | Denial of Service | `/v1` pre-auth chain (auth resolver + Postgres lookup) | mitigate | With T-01-01 closed, the coarse per-peer flood guard is effective again: a single peer cannot manufacture unlimited fresh buckets, so flood traffic is throttled before it reaches auth/DB (ROADMAP SC5 intent). |
| T-01-03 | Tampering | `jobs.client_id` attribution (`internal/jobs`) | mitigate | Tests must attribute jobs to a real `clients` row; the FK (`jobs_client_id_fkey`) rejects `uuid.Nil`/unknown clients (SQLSTATE 23503). Validated by the DATABASE_URL-backed `go test` in Task 1. |
| T-01-SC | Tampering (supply chain) | npm/pip/cargo/go installs | accept | No new packages are added. Both `github.com/go-chi/chi/v5` v5.3.0 and `github.com/go-chi/httprate` v0.15.0 are already pinned in go.mod/go.sum; `ClientIPFromRemoteAddr`/`GetClientIP` ship inside the already-vendored chi middleware package. Acceptance criteria assert go.mod/go.sum are unchanged. |

## Residual Risk (explicit, per security requirement)

`middleware.ClientIPFromRemoteAddr` is the correct choice **only because this service currently sits
behind no trusted reverse proxy** — internal clients connect directly (PROJECT.md constraints,
01-CONTEXT.md). The raw TCP peer address is therefore the real client identity and cannot be forged
at the application layer.

If a trusted L7 reverse proxy / load balancer is ever introduced in front of the API:
- `ClientIPFromRemoteAddr` would then key **every** client on the proxy's single IP (all clients share
  one bucket → over-throttling / shared-fate DoS), and real client IPs would become invisible to the
  guard.
- At that point the guard MUST switch to `middleware.ClientIPFromXFFTrustedProxies(numTrustedProxies)`
  (or an explicit trusted-CIDR configuration) plus a matching `GetClientIP`-based key func, so exactly
  the known-proxy hop count is stripped and no further. This is deliberately NOT built now (no proxy
  exists to trust); building speculative XFF-trust machinery today would re-open the very spoofing
  hole this plan closes. This residual is accepted and documented rather than mitigated.
</threat_model>
