<objective>
Give the image conversion queue its own retry budget and backoff schedule, replacing the confirmed defect where the server-wide `asynq.Config.RetryDelayFunc` (`queue.WebhookRetryDelay`) is silently applied to image tasks too, and where image tasks carry no `MaxRetry` option (defaulting to asynq's 25 attempts). Additionally, give image tasks a per-job uniqueness lock (`asynq.Unique`) whose TTL is derived from the actual retry budget (`IMAGE_MAX_RETRY` + `ENGINE_TIMEOUT`) so that no code path — including the Plan 03 reconciler's re-enqueue — can create a second, concurrent asynq task for a job whose task is still live.

Purpose: This is the queue-configuration half of retry-safety (RELY-02), plus the enabling half of RECON-01's "no duplicate tasks" guarantee. It makes asynq's own retry machinery use a fast, small, image-appropriate schedule so that once error classification (Plan 02) lands, transient image failures retry a few times quickly instead of either not retrying or grinding through 25 slow attempts on the wrong schedule; and it makes duplicate enqueues idempotent so the reconciler (Plan 03) can attempt recovery without ever double-processing a job whose original task is merely backlogged or still in-flight. NOTE: the derived TTL's soundness rests on Plan 02 bounding EVERY conversion attempt (download + convert + upload) by a single ENGINE_TIMEOUT deadline in process() — see 03-02 T-03-11; without that, a stalled S3 transfer could make one attempt exceed ENGINE_TIMEOUT and silently outlive this lock.

Output: A queue-aware `RetryDelayFunc` dispatcher, a fast `imageRetrySchedule` (2s/5s/15s), a configurable per-task `MaxRetry` for image tasks (`IMAGE_MAX_RETRY`, default 4) stored on `queue.Client`, a per-job `asynq.Unique` lock on image tasks whose TTL is derived from `IMAGE_MAX_RETRY` + `ENGINE_TIMEOUT` (so it can never fall behind asynq's true worst-case retry lifetime), and the worker server wired to the dispatcher.
</objective>

<phase_goal>
**As an** internal client submitting an image conversion, **I want to** have a temporary infrastructure hiccup retried a few times on a fast schedule, **so that** a transient blip does not translate into either a slow multi-minute retry loop or an accidental 25-attempt budget.

This plan delivers the observable behavior slice: image tasks retry on the fast 2s/5s/15s schedule with a bounded budget of `IMAGE_MAX_RETRY` (default 4), while webhook tasks keep their existing 30s->15m schedule; and a duplicate enqueue of a job whose task is still live is rejected (`asynq.ErrDuplicateTask`) rather than creating a second concurrent task.
</phase_goal>

So with IMAGE_MAX_RETRY=N a task gets N+1 total engine executions (1 initial + N retries) before archival — NOT N. The prior TTL math undercounted attempts by one; the derived ImageUniqueTTL below uses (maxRetry+1).
- VERIFIED asynq behavior: the unique lock is released on successful Done/MarkAsComplete or when its TTL expires; it is NOT released when a task is archived after exhausting retries (rdb.go archiveCmd does not touch the unique key). Therefore the image task's unique-lock TTL (derived by ImageUniqueTTL from IMAGE_MAX_RETRY + ENGINE_TIMEOUT) must exceed the maximum time asynq itself may legitimately keep retrying an image task, so the reconciler (Plan 03) defers to asynq while it is still retrying and only re-enqueues once the archived task's lock lapses. CRITICAL SOUNDNESS DEPENDENCY: this "worst-case = (maxRetry+1) * ENGINE_TIMEOUT + backoff" bound holds only if every single attempt is itself bounded by ENGINE_TIMEOUT. That per-attempt bound is enforced by Plan 02 (03-02 T-03-11), which wraps the WHOLE attempt (download + convert + upload + record) — not just conv.Convert() — in one context.WithTimeout(ctx, ENGINE_TIMEOUT). Do not weaken that assumption here.
