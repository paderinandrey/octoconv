<objective>
Add an operator-level `WEBHOOK_ALLOW_PRIVATE_IPS` opt-in that narrowly relaxes ONLY the RFC1918 private-IP check inside the webhook SSRF guard, so OctoConv deployments where internal clients live on private address space can actually receive webhooks — without weakening the default-safe posture for anyone else, and without loosening the loopback / link-local (cloud-metadata) / unspecified hard-blocks.

Purpose: Close the v1.0 tech-debt finding (WEBHOOK-06) where SSRF `callback_url` validation could make webhooks undeliverable on a real internal network.
Output: Conditional `isBlockedIP`, both-sides test coverage, a startup visibility warning, and `.env.example` documentation.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (job creation) | Client supplies `callback_url`; server later issues an outbound HTTP request to it (webhook delivery). Untrusted input crosses into server-initiated egress — the classic SSRF surface. |
| operator → deployment config | Operator sets `WEBHOOK_ALLOW_PRIVATE_IPS`; this changes the server's egress-target policy globally. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure (SSRF) | `isBlockedIP` in internal/api/callbackurl.go | mitigate | Default posture unchanged: with the flag unset, all RFC1918/loopback/link-local/unspecified targets stay blocked (Task 1 keeps the default branch identical; tests assert it). The opt-out is off-by-default (D-02). |
| T-05-02 | Elevation of Privilege (cloud-metadata SSRF, 169.254.169.254) | `isBlockedIP` | mitigate | D-01: only `addr.IsPrivate()` is gated on the flag. Link-local (which covers 169.254.0.0/16), loopback, and unspecified remain unconditional — asserted by TestIsBlockedIPAllowPrivate even with the flag on. No config value can unblock the metadata endpoint. |
| T-05-03 | Spoofing/Tampering (scheme downgrade) | `validateCallbackURL` | mitigate | The opt-out relaxes the IP-range check ONLY (SC3). http scheme stays gated by the separate `WEBHOOK_ALLOW_INSECURE_HTTP` flag; invalid/empty URLs still rejected. Task 1 behavior tests confirm `http://10.0.0.1/hook` still fails when only the private-IP flag is on. |
| T-05-04 | Repudiation (silent safety-posture change) | cmd/api/main.go startup | mitigate | D-04: startup emits a `⚠️` warning when the flag is enabled, so a relaxed posture is visible in logs without reading source or `.env`. |
| T-05-05 | Tampering (DNS rebinding — resolved host re-check) | `validateCallbackURL` | accept | Re-resolving `callback_url` before each delivery attempt remains explicitly out of scope (deferred, carried from Phase 2 D-03). Residual risk accepted: clients are internal-only services; enabling the flag is a deliberate internal-network operator choice. |
| T-05-SC | Tampering | dependency supply chain | accept | No new packages are added this phase (Go stdlib only: os, net/netip). No npm/pip/cargo/go-get install step; no Package Legitimacy Gate applies. |
