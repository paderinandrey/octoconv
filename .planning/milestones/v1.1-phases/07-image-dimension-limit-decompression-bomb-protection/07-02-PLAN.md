Wire the Phase 7 dimension check into the request path: add a configurable
`MAX_IMAGE_PIXELS` limit (default 100 megapixels) to the API config/env plumbing,
and call `convert.Dimensions` inside `handleCreateJob` between the pair-check and
callback_url validation to reject decompression-bomb-shaped uploads with 422
before any S3 write or job enqueue.

pixel count before the worker would decode it into gigabytes of memory. The check
must run inside the existing "reject before any storage write" gate sequence
(D-06), consistent with the Phase 4 content-validation discipline.

| T-07-04 | Denial of Service | `handleCreateJob` dimension gate | mitigate | Reject 422 when uint64(width)*uint64(height) > s.maxImagePixels BEFORE s.storage.Upload and queue.EnqueueImageConvert — no bomb reaches storage or the worker decode path. Verified by TestCreateJob_DimensionLimitExceeded asserting 422 + store.uploaded==false. |
| T-07-05 | Tampering / DoS | pixel-count product | mitigate | Product computed as uint64(width)*uint64(height); adversarial max-uint32 dimensions cannot wrap under the limit (Pitfall 1). Verified by grep gate + parser-level TestDimensionsOverflow (Plan 07-01). |
| T-07-07 | DoS (bypass via unparseable header) | dimensions_unknown path | mitigate | Fail closed: a Sniff-passing file whose declared dimensions cannot be located within the bounded window is rejected 422, not accepted (D-07). Verified by TestCreateJob_DimensionsUnknown. |
| T-07-08 | Information Disclosure | rejection responses | accept | Rejection messages state only the dimension-limit/unknown reason and filename; no internal error text leaked (consistent with existing handlers). Detailed width/height/limit go to the client_id-tagged internal log only (D-08), not the HTTP body. |
