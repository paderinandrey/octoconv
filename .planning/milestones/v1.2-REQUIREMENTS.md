# Requirements: OctoConv — v1.2 Document Engine Class

**Defined:** 2026-07-09
**Core Value:** Внутренние сервисы компании могут безопасно и надёжно поставить задачу конвертации файла и получить результат — без риска для стабильности или безопасности продакшена.

## v1 Requirements

Требования для milestone v1.2 (Document Engine Class). Каждое отображается на фазу roadmap.

### Document Content Validation

- [x] **DOC-01**: API принимает docx/xlsx/pptx/odt/ods/odp через существующий `POST /v1/jobs` и отклоняет несоответствие содержимого заявленному формату (структурный ZIP/OOXML/ODF sniff) с 422, до записи в S3 — по аналогии с magic-byte валидацией изображений (Phase 4)
- [x] **DOC-02**: API отклоняет документ, чей заявленный несжатый размер ZIP-контейнера превышает настраиваемый лимит (защита от zip-bomb), до конвертации
- [x] **DOC-03**: API отклоняет документ, содержащий части макросов (`vbaProject.bin` / Basic-script манифест), с 422

### LibreOffice Converter Engine

- [x] **DOC-04**: Воркер конвертирует принятые документы в PDF через LibreOffice headless с изолированным per-job профилем (`-env:UserInstallation`), исключающим коллизии lock-файла при параллельных job'ах
- [x] **DOC-05**: Воркер проверяет результат конвертации (ненулевой размер, валидные `%PDF-` magic bytes) перед пометкой задачи `done`; невалидный вывод — terminal-ошибка, задача не помечается успешной
- [x] **DOC-06**: Хардened process-exec (`Setpgid`+SIGKILL по таймауту) гарантированно завершает реальный `soffice`/`soffice.bin` процесс — подтверждено интеграционным тестом, а не предположением

### Queue, Worker & Reconciler Integration

- [x] **DOC-07**: Конвертация документов выполняется в отдельном воркер-процессе/контейнере (`cmd/document-worker`), изолированном по ресурсам (CPU/RAM) от image-воркера
- [x] **DOC-08**: Конвертация документов использует отдельный `DOCUMENT_ENGINE_TIMEOUT` (не общий `ENGINE_TIMEOUT`), таймаут классифицируется как terminal-ошибка
- [x] **DOC-09**: Reconciler корректно восстанавливает зависшие document-задачи через document-очередь (engine-aware), не через image-очередь

### API Routing & End-to-End Integration

- [x] **DOC-10**: `GET /v1/jobs/{id}`, webhook-доставка результата и presigned download URL работают для document-задач идентично image-задачам, без доработок (переиспользование существующей инфраструктуры)

## v2 Requirements

Отложено на будущее, не в этом roadmap.

### Document Engine — Extended

- **DOC-V2-01**: Кросс-конвертация внутри document-класса (docx↔odt, xlsx↔ods, pptx↔odp)
- **DOC-V2-02**: Pre-flight детекция password-protected legacy OLE-CFB документов как отдельная 422-причина
- **DOC-V2-03**: `opts`-driven PDF/A экспорт (плюмбинг `Convert(ctx, inPath, outPath, opts)` уже существует, не используется)
- **DOC-V2-04**: HTML → PDF через отдельный chromium-based движок (LibreOffice не подходит — слабый CSS/JS-рендеринг)
- **DOC-V2-05**: Активная защита от ресурсного DoS через сложность документа (лимит листов/ячеек/размера распакованного ZIP) — v1.2 принимает это остаточным риском, ограниченным таймаутом + отдельным concurrency-лимитом на document-очередь

## Out of Scope

Явно исключено. Задокументировано, чтобы предотвратить scope creep.

| Feature | Reason |
|---------|--------|
| CAD-движок | Открытый вопрос по SDK (OSS vs коммерческий vs cloud API), не решён — отложен |
| Другие классы движков (av/ffmpeg, archive, probe) | Следующий этап роста, не этот milestone |
| Полный контракт ядра (Handler/Capability/Input/Output/Progress) | Второй движок укладывается в существующий `Converter`/`Registry` без рефакторинга |
| Watermarking, full OOXML/ODF schema validation | Java-based тулинг конфликтует с zero-new-deps философией; watermarking конфликтует с posture "отключить макросы/внешний контент" |
| Persistent/pooled LibreOffice listener (`unoconv`, `soffice --accept=socket`) | Решает проблему конкурентности, которой при per-invocation модели не существует; конфликтует с "no core rework" |
| Активный anti-DoS по сложности документа (сверх таймаута) | Признано остаточным риском в v1.2 (см. DOC-V2-05), как ранее CAD/HTML→PDF |
