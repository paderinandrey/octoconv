# Phase 9: LibreOffice Converter Engine - Research

**Researched:** 2026-07-09
**Domain:** Second `convert.Converter` implementation shelling out to LibreOffice headless (`soffice`), reusing the existing hardened `os/exec` wrapper, in a Go async job-processing worker
**Confidence:** HIGH — every specific technical unknown listed in the research brief was resolved empirically against a live `debian:bookworm-slim` container built with the exact recommended package set (Docker was available in this research environment), not just cited from milestone-level research or training knowledge.

## Summary

This is a deep-dive, verification-focused research pass on top of already-solid milestone-level research (STACK.md/PITFALLS.md/SUMMARY.md). Every one of the six specific technical unknowns flagged in the research brief was resolved with a live Docker test against `debian:bookworm-slim` today (2026-07-09), using the exact package set milestone research recommended. The headline results: (1) the package set installs cleanly and is still current (`4:7.4.7-1+deb12u13`, verified via `apt-cache policy` inside the built container, zero important unpatched CVEs as of today); (2) the process-group-kill safety net **is proven to work** — `/usr/bin/soffice` is a shell script that `exec`s into `oosplash`, which **forks** (not execs) `soffice.bin` as a child that inherits the same process group, so `runCommand`'s existing `Setpgid`+group-`SIGKILL` kills both the launcher and the real conversion process with zero survivors, confirmed while `soffice.bin` was actively mid-conversion on a 5MB document; (3) per-job `-env:UserInstallation` isolation gives genuine concurrency — 4 simultaneous conversions with distinct profile dirs succeeded independently with no lock collisions; (4) the `Converter` interface needs **zero changes** — `LibreOfficeConverter.Convert` can self-derive its isolation directory as a subdirectory of `filepath.Dir(outPath)`, because `internal/worker/worker.go`'s `process()` already places `inPath` and `outPath` inside the same per-job `os.MkdirTemp` `workDir`; (5) `USER nobody` with an unwritable `$HOME` does **not** break conversion — it degrades gracefully (stderr warnings about fontconfig/dconf cache, but exit 0 and a correct PDF every time) — contrary to the milestone research's more alarmist "cryptic failures" framing, though setting `HOME` to the per-job workDir is still recommended to eliminate the warning noise and a redundant per-invocation font-cache rebuild; (6) exact filter names and the outdir-basename-to-`in.pdf` rename convention were confirmed for all 6 source formats using real (not synthetic) office documents generated by LibreOffice itself from its own bundled templates.

**Primary recommendation:** Implement `internal/convert/libreoffice.go` as a small, stateless `LibreOfficeConverter` struct mirroring `libvips.go`'s shape exactly — `Pairs()` returns the 6 `{format, "pdf"}` pairs, `Convert()` derives an isolated profile dir from `filepath.Dir(outPath)`, invokes `runCommand` (unchanged) with an explicit per-module PDF export filter, renames LibreOffice's `in.pdf` output to the caller's `outPath`, and validates the result (non-zero size + `%PDF-` magic bytes) before returning success. Add the LibreOffice `-nogui` packages + font packages to `Dockerfile.worker` exactly as milestone research specified — no changes needed. Ship a live, binary-presence-gated Go integration test (mirroring the project's existing `DATABASE_URL`-gated soak-test pattern) that proves the process-group-kill guarantee against a real hung `soffice.bin`, per D-03.

### Pattern 1: Self-Contained Profile Isolation (no `Converter` interface change)

**What:** `LibreOfficeConverter.Convert` derives its own per-invocation `-env:UserInstallation` directory from `filepath.Dir(outPath)`, rather than requiring a new parameter or interface change.
**When to use:** Always, for this converter — this is the resolution to research brief unknown #6.
**Why it's correct:** `internal/worker/worker.go:270-278` (read directly) already creates one `os.MkdirTemp` `workDir` per job and places both `inPath` and `outPath` inside it before calling `conv.Convert(ctx, inPath, outPath, nil)`. `filepath.Dir(outPath)` is therefore already unique-per-job and already cleaned up by the caller's `defer os.RemoveAll(workDir)` — no new lifecycle code needed.
**Example (Go, matches project conventions — `libvips.go` structural analog):**
```go
// Source: derived from internal/worker/worker.go:270-278 (read directly) +
// live-verified concurrency test (4 simultaneous jobs, zero collisions), 2026-07-09
func (LibreOfficeConverter) Convert(ctx context.Context, inPath, outPath string, _ map[string]any) error {
    workDir := filepath.Dir(outPath) // caller's per-job workDir; already unique, already cleaned up
    profileDir := filepath.Join(workDir, "lo-profile")
    if err := os.MkdirAll(profileDir, 0o700); err != nil {
        return fmt.Errorf("libreoffice: mkdir profile: %w", err)
    }

    filter, err := filterFor(filepath.Ext(inPath))
    if err != nil {
        return fmt.Errorf("libreoffice: %w", err)
    }

    args := []string{
        "--headless", "--invisible", "--nocrashreport", "--nodefault",
        "--nologo", "--nofirststartwizard", "--norestore",
        "-env:UserInstallation=file://" + profileDir,
        "--convert-to", "pdf:" + filter,
        "--outdir", workDir,
        inPath,
    }
    if err := runCommand(ctx, "soffice", args...); err != nil {
        return fmt.Errorf("libreoffice: %w", err)
    }

    // soffice writes <input-basename>.pdf into --outdir; the worker's inPath
    // convention (internal/worker/worker.go) is always "in.<sourceFormat>",
    // so the produced file is deterministically workDir/in.pdf — VERIFIED
    // live for all 6 source formats. Rename it to the caller's outPath.
    producedPath := filepath.Join(workDir, strings.TrimSuffix(filepath.Base(inPath), filepath.Ext(inPath))+".pdf")
    if err := os.Rename(producedPath, outPath); err != nil {
        return fmt.Errorf("libreoffice: rename output: %w", err)
    }

    return validatePDF(outPath)
}
```

### Pattern 2: Explicit Per-Module PDF Export Filter

**What:** Pin `writer_pdf_Export` / `calc_pdf_Export` / `impress_pdf_Export` explicitly rather than relying on `--convert-to pdf`'s auto-detection.
**Verified live (2026-07-09):** real fixtures generated by LibreOffice itself (from `/usr/lib/libreoffice/share/template/...`) for all 6 formats — docx, odt (→ `writer_pdf_Export`), xlsx, ods (→ `calc_pdf_Export`), pptx, odp (→ `impress_pdf_Export`) — produced **byte-identical PDFs** (same size, same `%PDF-1.6` header) whether the filter was pinned explicitly or left to auto-detection. Pin explicitly anyway per milestone research's rationale (documents intent, removes ambiguity for future format additions).
```go
// Source: LibreOffice Help - File Conversion Filter Names (official docs) +
// live-verified against real docx/xlsx/pptx/odt/ods/odp fixtures, 2026-07-09
func filterFor(sourceExt string) (string, error) {
    switch convert.NormalizeFormat(sourceExt) {
    case "docx", "odt":
        return "writer_pdf_Export", nil
    case "xlsx", "ods":
        return "calc_pdf_Export", nil
    case "pptx", "odp":
        return "impress_pdf_Export", nil
    default:
        return "", fmt.Errorf("no pdf export filter for %q", sourceExt)
    }
}
```

### Pattern 3: Generic PDF Output Validation (magic bytes + size)

**What:** Non-zero size AND first 5 bytes equal `%PDF-`, mirroring `sniff.go`'s `matchPNG`/`matchJPEG` style.
**Locked placement (CONTEXT.md D-02):** lives **inside** `LibreOfficeConverter.Convert`, not as a generic hook in `worker.go`'s `process()`. This differs from milestone PITFALLS.md Pitfall 2, which suggested a generic engine-agnostic check in `process()` that would also silently harden the image path — CONTEXT.md's locked decision D-02 explicitly assigns this responsibility to the converter itself. Follow CONTEXT.md; do not resurrect the milestone suggestion without a discuss-phase revisit.
```go
// Source: mirrors internal/convert/sniff.go's matchPNG-style magic-byte check
// (project's established validation idiom), verified against real %PDF-1.6
// output produced live, 2026-07-09.
var pdfMagic = []byte("%PDF-")

func validatePDF(path string) error {
    fi, err := os.Stat(path)
    if err != nil {
        return fmt.Errorf("libreoffice: stat output: %w", err)
    }
    if fi.Size() == 0 {
        return fmt.Errorf("libreoffice: output is empty")
    }
    f, err := os.Open(path)
    if err != nil {
        return fmt.Errorf("libreoffice: open output: %w", err)
    }
    defer f.Close()
    buf := make([]byte, len(pdfMagic))
    if _, err := io.ReadFull(f, buf); err != nil {
        return fmt.Errorf("libreoffice: read output header: %w", err)
    }
    if !bytes.Equal(buf, pdfMagic) {
        return fmt.Errorf("libreoffice: output missing %%PDF- magic bytes")
    }
    return nil
}
```

### Anti-Patterns to Avoid

- **Sharing a `-env:UserInstallation` directory across jobs:** confirmed to be the concurrency-collision root cause in milestone research; per-job isolation via `filepath.Dir(outPath)` is a one-time, zero-cost fix.
- **Assuming `HOME` must be writable for correctness:** empirically false — conversion succeeds with exit 0 and a correct PDF even with `HOME=/nonexistent`. Do not block the plan on solving this as a correctness bug; treat it as a log-cleanliness/performance nicety (see Common Pitfalls).
- **Assuming build-time `fc-cache -f` (as root) eliminates the fontconfig warning:** empirically false — the warning is about `$HOME/.cache/fontconfig` (a user-level cache), not `/var/cache/fontconfig` (the system cache `fc-cache -f` populates). Prebuilding the system cache did not remove the warning in live testing.
- **Treating stderr output as inherently a failure signal:** LibreOffice emits `dconf-CRITICAL` and `Fontconfig error` lines on every successful conversion under `USER nobody` with no `HOME` override — these are **not** failures (exit 0, correct output). A future `terminalLibreOfficeSignatures` list (mirroring `terminalVipsSignatures`) must not naively match on generic words like "error" or "critical" — it must match specific, documented failure phrases only.
- **Adding a `unoconv`/persistent listener:** rejected by milestone research and reconfirmed unnecessary — per-invocation concurrency already works.

**Key insight:** Every piece of this phase's implementation is either already-existing project infrastructure (`runCommand`, the per-job `workDir`, the `sniff.go` magic-byte idiom) reused unchanged, or a small, explicit, verifiable mapping (filter names). Nothing in DOC-04/05/06 requires new abstractions.

## Common Pitfalls

### Pitfall 1: Exit-0-but-corrupt output (still a real, documented LibreOffice failure mode)
**What goes wrong:** LibreOffice has a well-documented history (TDF bugzilla #52125) of exiting 0 while producing an empty or truncated PDF.
**Why it happens:** Internal export-filter failures don't always propagate to the process exit code.
**How to avoid:** The size + `%PDF-` magic-bytes check (Pattern 3) is the direct, locked (D-02) mitigation. Not reproduced in this session's live tests (every conversion of well-formed input succeeded correctly) — this remains a documented-but-not-personally-observed risk; the validation check is cheap insurance regardless.
**Warning signs:** Any `done` document job whose output size is anomalously small.

### Pitfall 2: Process-group-kill gap — RESOLVED, empirically closed
**What was feared:** A forking launcher (`oosplash`) could detach `soffice.bin` into a new session, escaping `Setpgid`+group-`SIGKILL`.
**What was found (empirical, 2026-07-09):** `/usr/bin/soffice` → `exec`s `oosplash` (same PID/PGID) → `oosplash` **forks** `soffice.bin` as a child. Neither process calls `setsid`/`setpgid` to detach — `soffice.bin` inherits `oosplash`'s process group by default POSIX fork semantics. Verified via a live probe: launched `soffice --headless` on a 5MB text document (wide enough render window to catch it mid-flight), polled `ps -eo pid,ppid,pgid,stat,comm` every 20ms until `soffice.bin` was observed in state `R` (actively running), then sent `SIGKILL` to the negative process group. Result: **zero surviving `soffice`/`soffice.bin`/`oosplash` processes**, confirmed immediately and again 2s later; output directory was empty (proving the kill happened genuinely mid-conversion, not after natural completion).
**Conclusion:** `internal/convert/exec.go`'s existing `runCommand` requires **no changes**. D-03's integration test (see Code Examples below) should encode exactly this probe-and-kill methodology.
**Residual risk:** This was verified against `4:7.4.7-1+deb12u13` on `debian:bookworm-slim` (arm64) specifically — a future Debian point-release could in principle change the launcher topology. This is precisely why D-03 mandates a live, repeatable integration test rather than treating this research finding as permanent.

### Pitfall 3: `USER nobody` + unwritable `$HOME` — degrades gracefully, does NOT hang or crash (refines milestone PITFALLS.md)
**What was feared (milestone research):** "cryptic, not obviously LibreOffice-related, startup failures... hanging on startup... a bare non-descriptive fatal crash."
**What was found (empirical, 2026-07-09):** Running as `USER nobody` with `HOME=/nonexistent` and only `-env:UserInstallation` set (no `HOME` override) produced:
```
(process:26): dconf-CRITICAL **: unable to create directory '/nonexistent/.cache/dconf': Permission denied.
Fontconfig error: No writable cache directories
convert /tmp/job1/in.docx -> /tmp/job1/out/in.pdf using filter : writer_pdf_Export
exit code: 0
```
A correct, valid PDF was still produced (matching byte-for-byte the version produced with `HOME` set to a writable directory). **This is not launch-blocking** — the milestone research's framing overstated the risk.
**Refinement — build-time `fc-cache -f` does NOT fix this:** rebuilding the image with `RUN fc-cache -f` (as root, populating `/var/cache/fontconfig`) did **not** eliminate the warning on a subsequent `USER nobody` run — the warning is specifically about `$HOME/.cache/fontconfig` (a per-user cache fontconfig always attempts), not the system cache directory `fc-cache -f` populates. A planner following milestone PITFALLS.md's Pitfall 5 literally (prebuild fontconfig cache as root) would ship a Dockerfile change that does not actually resolve the symptom it targets.
**Recommended mitigation (still worth doing, but for cleanliness/performance, not correctness):** set `cmd.Env` (or pass `HOME=` via `exec.Command`'s `Env` field) to the same per-job `workDir` already available in `Convert()` — this eliminates the stderr noise entirely and avoids rebuilding the in-memory font cache on every single invocation (a real, if unmeasured, per-job latency cost).
**Warning signs to explicitly NOT alarm on:** `dconf-CRITICAL` and `Fontconfig error: No writable cache directories` lines are benign when they appear alongside `exit code: 0` and a valid output file — do not build a naive stderr-substring terminal-classifier that treats "error"/"critical" as a failure signal.

### Pitfall 4: Profile-lock concurrency — RESOLVED, empirically closed
**What was feared:** Concurrent jobs sharing a profile directory would collide.
**What was found (empirical, 2026-07-09):** 4 simultaneous `soffice --headless` invocations, each with a distinct `-env:UserInstallation` directory (mirroring the per-job `filepath.Dir(outPath)` derivation), all completed successfully in parallel with correct, identical-sized output and zero lock errors in any of the 4 logs.
**Conclusion:** Per-job isolation (Pattern 1) is sufficient; no additional concurrency ceiling is needed at the converter level for DOC-04/05/06's scope (Phase 10 still owns the separate question of the `document` queue's overall worker concurrency limit, per milestone research).

### Pitfall 5: `opts map[string]any`, not `map[string]string` — signature discrepancy in the research brief
**What goes wrong:** The research brief's unknown #6 referenced `Convert(ctx, inPath, outPath string, opts map[string]string) error`. The actual interface in `internal/convert/convert.go:24` is `opts map[string]any`.
**How to avoid:** `LibreOfficeConverter.Convert`'s signature must use `map[string]any` (or `_ map[string]any` if unused, matching `LibvipsConverter.Convert`'s exact style) — using `map[string]string` will fail to satisfy the `Converter` interface at compile time.

### Pitfall 6 (non-blocking, accepted): Font-substitution / version-skew output variance
Unchanged from milestone PITFALLS.md's Pitfall 7 — accept as a documented limitation, not an engineering problem. Not re-litigated here since milestone research already covers it adequately and no new information was found this session.

## Code Examples

### D-03 Live Integration Test (process-group-kill proof)

```go
// Source: pattern mirrors internal/reconciler/reconciler_soak_test.go's
// DATABASE_URL-gated skip idiom, adapted to gate on soffice binary presence
// (this test needs a real installed LibreOffice, only present inside the
// worker Docker image / CI, not a bare `go test` dev environment).
// Methodology verified live, 2026-07-09: poll ps for soffice.bin in state R,
// SIGKILL the process group, assert zero survivors.
package convert

import (
	"bytes"
	"context"
	"os"
	"os/exec"
	"strings"
	"testing"
	"time"
)

func TestLibreOfficeConverter_TimeoutKillsRealProcess(t *testing.T) {
	if _, err := exec.LookPath("soffice"); err != nil {
		t.Skip("soffice not on PATH; skipping (run inside the worker Docker image or CI)")
	}

	// A large-enough input widens the conversion window so the timeout can
	// fire while soffice.bin is genuinely still rendering, not after it has
	// already exited naturally.
	dir := t.TempDir()
	inPath := dir + "/in.txt"
	var buf bytes.Buffer
	for i := 0; i < 80_000; i++ {
		buf.WriteString("the quick brown fox jumps over the lazy dog\n")
	}
	if err := os.WriteFile(inPath, buf.Bytes(), 0o600); err != nil {
		t.Fatal(err)
	}

	ctx, cancel := context.WithTimeout(context.Background(), 50*time.Millisecond)
	defer cancel()

	outPath := dir + "/out.pdf"
	_ = LibreOfficeConverter{}.Convert(ctx, inPath, outPath, nil) // expected to time out

	out, err := exec.Command("ps", "-eo", "pid,comm").CombinedOutput()
	if err != nil {
		t.Fatalf("ps: %v", err)
	}
	for _, line := range strings.Split(string(out), "\n") {
		if strings.Contains(line, "soffice") || strings.Contains(line, "oosplash") {
			t.Fatalf("surviving LibreOffice process after timeout: %s", line)
		}
	}
}
```
