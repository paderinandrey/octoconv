<objective>
Gap-closure for Phase 36 code-review findings CR-01 (CRITICAL) and HI-01 (HIGH). The phase's headline security fix (disposition (b), "bound the passthrough path", closing the measured hevc@2160p OOM-KILL DoS) was implemented for the no-scale (`resolution_height==0`) path ONLY. An explicit `resolution_height` request from an oversized source, and any extreme-WIDTH source, both bypass the memory-safety bound and plausibly reproduce the same OOM/timeout class — while the .env/compose docs advertise the vector as closed. This plan generalizes the bound to EVERY re-encode path on BOTH axes, flips the test that asserted the bypass as intended, and corrects the docs so the claim is true. Stream-copy remux stays exempt.
</objective>

<context>
Root cause: the OOM/RTF cost is driven by DECODING the source; the RTF matrix only ever measured sources generated AT the target resolution (no decode-then-downscale cell), so the derived 90s/753s/1g envelope only covers sources ≤1080p. Therefore any re-encode of a source above 1920x1080 is unmeasured and unbounded. Fail-closed rejection to the measured envelope is the correct, shippable disposition-(b) completion; measuring a larger decode-then-downscale envelope is deferred future work (name it, don't silently allow it).

Existing anchors (internal/convert/av.go):
- avMaxSourceResolutionHeight = 4320 (decode-bomb ceiling, Height-only, EnforceMaxResolution) — the outer guard.
- avNoScalePassthroughMaxHeight = 1080; enforceNoScalePassthroughBound(height) — the no-scale-only bound to generalize.
- Width is probed on avVideoStream (.Width) but bounded nowhere.
- The re-encode vs stream-copy split is avStreamCopyEligible; the guard must sit on the decode/encode branch only.
- Terminal/non-retryable classification: a too-large source is a deterministic client error (like ErrAVResolutionExceeded) — classify accordingly.
</context>
