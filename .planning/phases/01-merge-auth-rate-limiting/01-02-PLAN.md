<objective>
Turn issued keys into enforced authentication: build the `internal/auth` resolver + chi middleware, wire it onto the `/v1` route group, thread the authenticated `client_id` through job creation, and enforce per-client ownership on job reads (404 for cross-client access). This is the "every request is authenticated and client-scoped" vertical slice — a valid key yields 200, everything else yields 401, and one client can never observe another's jobs.

Purpose: closes PITFALLS Pitfall 5 (auth trusting network position) and the AUTH-03 enumeration leak. Hard cutover per D-08 — no warn-only path.
Output: `internal/auth` (auth.go, context.go, middleware.go), job `client_id` threading, wired `/v1` group, cross-client 404.
</objective>

    - Middleware with a resolver that returns a client: request passes through; the handler can read the client via ClientFromContext (ok=true) and gets the same client.ID.
    - Missing Authorization header → 401, next handler NOT invoked.
    - Authorization header without the "ApiKey " scheme (e.g. bare token or "Bearer x") → 401.
    - Authorization "ApiKey <key>" where resolver returns ErrInvalidKey → 401, next NOT invoked.
    - Resolver returning a non-ErrInvalidKey error (e.g. DB down) → 500, next NOT invoked.
    - 401/500 responses have Content-Type application/json and a JSON body with an "error" field; the response never echoes the presented key.
    - Resolver.ResolveClient hashes the raw key via HashKey(salt, raw) and maps clients.ErrNotFound → auth.ErrInvalidKey.

Define `var ErrInvalidKey = errors.New("invalid api key")`. Implement `type Resolver struct { store keyStore; salt []byte }`, `func NewResolver(store keyStore, salt []byte) *Resolver`, and `func (r *Resolver) ResolveClient(ctx, rawKey string) (*clients.Client, error)`: compute `h := HashKey(r.salt, rawKey)`, call `r.store.GetByKeyHash(ctx, h)`, and translate `clients.ErrNotFound` → `ErrInvalidKey` (wrap other errors with `fmt.Errorf("resolve client: %w", err)`). Note in a comment: the raw key is reduced to a salted SHA-256 digest before the only equality check (an indexed lookup in Postgres), so no variable-time comparison of raw secret material occurs in Go.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (/v1/*) | Untrusted Authorization header + client-supplied job id cross here; must be authenticated and ownership-scoped |
| resolved client identity in request context | Downstream handlers trust ClientFromContext; it must only ever be set by the auth middleware after a successful resolve |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Spoofing | /v1 route group | mitigate | Mandatory `auth.Middleware` on the /v1 group; missing/malformed/invalid/revoked key → 401 before any handler; hard cutover (D-08), no warn/skip path |
| T-01-05 | Information disclosure | handleGetJob (IDOR/enumeration) | mitigate | Ownership check returns 404 with the identical body as true-not-found for another client's job — never 403, never a distinct message (AUTH-03) |
| T-01-06 | Elevation of privilege | routing / network trust | mitigate | Auth enforced by middleware on the router, not per-handler opt-in; only `/healthz` is an explicit public allowlist (D-09); no network-position or env "skip auth" toggle exists |
| T-01-07 | Information disclosure (timing) | key comparison | accept | Raw key is reduced to a salted SHA-256 digest before the sole equality check (indexed Postgres lookup); no variable-time Go comparison of raw secret occurs. `crypto/subtle`/`hmac.Equal` deferred to Phase 2 HMAC webhook verification where symmetric in-memory comparison actually happens |
| T-01-SC | Tampering | go.mod dependencies | accept | Plan 02 adds no new module dependency (stdlib net/http, strings, context only) |
</threat_model>
