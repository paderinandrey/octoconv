<objective>
Protect the service from abuse: add `github.com/go-chi/httprate`, build an `internal/ratelimit` package exposing a coarse pre-auth IP limiter and a fair per-client limiter, and slot them into the `/v1` middleware chain in the research-mandated order (coarse IP → auth → per-client). This is the "excessive traffic is rejected" vertical slice — a client over its rate gets 429 + Retry-After; a flood of garbage keys is throttled before it touches auth or the DB.

Purpose: closes PITFALLS Pitfall 9 (rate limiting on the wrong identity) and Anti-Pattern 3 (no pre-auth guard). Per-client limit MUST key on the authenticated client_id, so it runs after auth.
Output: `internal/ratelimit`, wired /v1 chain, env-configurable thresholds.
</objective>

| authenticated client → /v1 handlers | A single client must not monopolize capacity; limit keyed on verified identity, not spoofable IP |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-01-08 | Denial of service | pre-auth flood → auth/DB | mitigate | `ratelimit.ByIP` runs FIRST on /v1, before auth, so garbage-key floods are throttled before any Postgres lookup (RATE-03; PITFALLS Anti-Pattern 3) |
