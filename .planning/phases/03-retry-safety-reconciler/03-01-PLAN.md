<objective>
Give the image conversion queue its own retry budget and backoff schedule, replacing the confirmed defect where the server-wide `asynq.Config.RetryDelayFunc` (`queue.WebhookRetryDelay`) is silently applied to image tasks too, and where image tasks carry no `MaxRetry` option (defaulting to asynq's 25 attempts). Additionally, give image tasks a per-job uniqueness lock (`asynq.Unique`) so that no code path — including the Plan 03 reconciler's re-enqueue — can create a second, concurrent asynq task for a job whose task is still live.

Purpose: This is the queue-configuration half of retry-safety (RELY-02), plus the enabling half of RECON-01's "no duplicate tasks" guarantee. It makes asynq's own retry machinery use a fast, small, image-appropriate schedule so that once error classification (Plan 02) lands, transient image failures retry a few times quickly instead of either not retrying or grinding through 25 slow attempts on the wrong schedule; and it makes duplicate enqueues idempotent so the reconciler (Plan 03) can attempt recovery without ever double-processing a job whose original task is merely backlogged or still in-flight.

Output: A queue-aware `RetryDelayFunc` dispatcher, a fast `imageRetrySchedule` (2s/5s/15s), a configurable per-task `MaxRetry` for image tasks (`IMAGE_MAX_RETRY`, default 4) stored on `queue.Client`, a per-job `asynq.Unique` lock on image tasks, and the worker server wired to the dispatcher.
</objective>

<phase_goal>
**As an** internal client submitting an image conversion, **I want to** have a temporary infrastructure hiccup retried a few times on a fast schedule, **so that** a transient blip does not translate into either a slow multi-minute retry loop or an accidental 25-attempt budget.

This plan delivers the observable behavior slice: image tasks retry on the fast 2s/5s/15s schedule with a bounded budget of `IMAGE_MAX_RETRY` (default 4), while webhook tasks keep their existing 30s->15m schedule; and a duplicate enqueue of a job whose task is still live is rejected (`asynq.ErrDuplicateTask`) rather than creating a second concurrent task.
</phase_goal>
