<objective>
Introduce transient-vs-terminal error classification in `HandleImageConvert` and make `MarkActive` idempotent for `active -> active` re-entry, so asynq's retry mechanism actually takes effect for transient image failures while terminal failures fail fast — and sanitize the stored `error_message` so raw engine stderr (which contains local temp file paths) is not exposed via `GET /jobs/{id}` or webhook payloads.

Purpose: This is the worker/repository half of retry-safety (RELY-01, RELY-02) and the core bugfix. Currently `HandleImageConvert` calls `MarkFailed` unconditionally on any error, and `MarkActive` allows only `queued -> active`, so every asynq-internal retry of the same task fails at the first line and gets `SkipRetry`-wrapped — every job gets exactly one real attempt regardless of retry config.

Output: an `isTerminal` classifier, a two-branch error handler mirroring the existing `HandleWebhookDeliver` "unwrap error -> let asynq retry" pattern, a widened idempotent `MarkActive`, a `transition()`/`MarkFailed` that can attach a `job_events.detail` payload, and sanitized failure messages.
</objective>

<phase_goal>
**As an** internal client submitting an image conversion, **I want to** have transient failures (network, timeout, temporary S3/Postgres errors) retried automatically while genuinely broken inputs fail immediately, **so that** a temporary hiccup self-heals and a corrupt file does not waste a retry budget or leak internal file paths back to me.

This plan delivers the observable behavior slice: inject a transient failure -> the job stays `active` and asynq re-runs the engine; inject a terminal failure (corrupt input / unsupported pair) -> the job is `failed` with a short, sanitized reason.
</phase_goal>
