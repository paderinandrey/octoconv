<objective>
Add a Postgres-driven reconciler that periodically sweeps for jobs stranded in `queued` (lost enqueue) or `active` (crashed worker / exhausted asynq retry budget) past staleness thresholds, requeues them idempotently through a guarded transition, caps the number of recoveries per job, and terminally fails jobs that exceed the cap — recording every action in `job_events` and firing a webhook on exhaustion. The re-enqueue is guarded by the per-job `asynq.Unique` lock added in Plan 01: the reconciler attempts the enqueue first and only records a recovery when a genuinely new task is created, so a merely-backlogged or still-in-flight job is never double-processed and never falsely exhausted.

Purpose: RECON-01/02/03. The research established this is a REQUIRED backstop the moment retry-safety (Plans 01/02) lands: once transient failures leave a job `active` and asynq silently archives the task after its own small retry budget, only the reconciler can eventually flip such a job to a resolved state. Postgres (not Redis/asynq Inspector) is the source of truth for staleness; the asynq uniqueness lock (Plan 01) is the source of truth for "is a task still live," letting the reconciler avoid duplicate work without an O(n) Inspector scan.

Output: new `RequeueStale`/`RecoveryCount`/`FindStale` repository methods, a new `internal/reconciler` package with a ticker-driven `Sweeper` whose recovery path is guarded by `asynq.ErrDuplicateTask`, and worker-process wiring with coordinated graceful shutdown.
</objective>

<phase_goal>
**As an** operator of OctoConv, **I want to** have jobs stranded by a lost enqueue or a crashed worker automatically recovered (and permanently-broken ones terminally failed), **so that** no job silently sits stuck forever and clients always eventually get a done or failed outcome without manual intervention — and without duplicating work.

This plan delivers the observable behavior slice: a job left in `queued`/`active` past its threshold, whose task is genuinely gone, gets re-enqueued and reprocessed exactly once; a job whose task is merely backlogged is left alone; after N failed recoveries a job is marked `failed` (error_code reconciler_exhausted) and — if a callback_url is set — a webhook is delivered.
</phase_goal>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Tampering / Repudiation | reconciler status changes | mitigate | All recovery/exhaustion status changes route through Repo.transition's SELECT ... FOR UPDATE row lock (via RequeueStale/MarkFailed); no ad-hoc UPDATE. Every action is logged to job_events (RECON-03), providing a non-repudiable audit trail |
| T-03-07 | Denial of Service | permanently-broken job retried forever | mitigate | RecoveryCount cap (RECONCILER_MAX_RECOVERIES, default 3) terminally fails a job as reconciler_exhausted after N recoveries — bounds repeated engine invocation. RequeueStale (which increments the count) runs ONLY after a genuine non-duplicate enqueue, so a merely-backlogged job cannot silently accrue recoveries toward this cap; and because a lost RequeueStale AFTER a real enqueue would under-count the cap the OTHER way, RequeueStale is wrapped in a bounded single retry with an explicit-comment discard path, so cap accounting degrades gracefully (loss bounded to one event per sweep) rather than silently under-counting a permanently-broken job past MaxRecoveries |
| T-03-08 | Elevation of Privilege / Info Disclosure | reconciler operates on all jobs (not client-scoped) | accept | Matches the existing internal-only trust model; reconciler is a backend process with no external caller, and RECON-03 audit entries carry no new sensitive data (raw stderr stays in job_events.detail only, per Plan 02) |
| T-03-09 | Tampering | reconciler vs legitimately-slow worker race (Pitfall 4) | accept | Documented, accepted MVP limitation (STATE.md blocker + SCALE-V2-03 deferred). The row lock ensures the worst case (double-commit) fails loudly: a slow worker's MarkDone on a requeued job fails the guarded transition rather than corrupting state. Mitigated by RECONCILER_ACTIVE_STALE_AFTER (5m) set well above ENGINE_TIMEOUT (120s) and, additionally, by the asynq.Unique lock (T-03-10) which prevents the reconciler from even creating a competing task while the slow worker's task is still active-and-unarchived |
| T-03-10 | Tampering / Denial of Service | reconciler duplicating an already-in-flight task -> concurrent double-processing | mitigate | RECON-01's "no duplicate tasks" is enforced by two coupled mechanisms: (1) Plan 01 gives every image task an asynq.Unique lock keyed by queue+type+job-id, so EnqueueImageConvert returns asynq.ErrDuplicateTask whenever a task/lock for the job is still live (backlogged, scheduled, retrying, or active-and-not-yet-archived); (2) the sweep enqueues FIRST and treats ErrDuplicateTask as a no-op (no status flip, no recovery event), only calling RequeueStale after a genuinely new task is created. Together these prevent a second concurrent process() invocation, duplicate S3 writes, and the terminal-transition race that the Plan-02 widening of MarkActive (queued|active) would otherwise expose — and prevent a backlogged queued job (burst exceeding WORKER_CONCURRENCY past the 90s threshold) from being falsely driven to reconciler_exhausted. This guarantee holds only while the unique-lock TTL genuinely outlives every legitimate in-flight lifetime: Plan 01's ImageUniqueTTL derives it from IMAGE_MAX_RETRY + ENGINE_TIMEOUT (~757s for the defaults, > asynq's 637s worst-case retry lifetime) AND Plan 02 (T-03-11) bounds each whole conversion attempt (download + convert + upload + record) by a single ENGINE_TIMEOUT deadline, so a stalled S3 transfer cannot make one attempt exceed ENGINE_TIMEOUT, outlive the lock mid-flight, and let this reconciler create a duplicate. Residual: a single job continuously occupying the queue longer than the derived TTL falls into the same accepted lease-staleness envelope as T-03-09 |
| T-03-12 | Tampering / Repudiation | bounded RequeueStale retry double-writing one recovery event (post-commit/pre-ack transport error) | accept | If the FIRST RequeueStale commits server-side but the client observes a transport error between COMMIT and ack (a case pgx.BeginFunc does not exclude), the bounded single retry executes a SECOND genuinely-successful transition and inserts a SECOND reconciler_recovery job_events row for ONE real recovery — over-counting RecoveryCount by one, so the job may reach reconciler_exhausted one recovery earlier than intended. Bounded (exactly one extra event, since the retry is single-shot), low-probability (requires a transport failure inside the narrow COMMIT->ack window), and does NOT reopen the no-duplicate-task guarantee (independently protected by asynq.Unique, T-03-10). Symmetric to the under-count path already handled by the bounded retry in T-03-07 |
