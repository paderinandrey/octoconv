## Классификация ошибок: transient vs terminal

| Option | Description | Selected |
|--------|-------------|----------|
| Широкий retry | Terminal только: нет конвертера для пары форматов, движок явно сообщает bad format/corrupted input. Всё остальное — transient. | ✓ |
| Консервативный retry | Любой ненулевой exit code libvips = terminal по умолчанию; retry только для storage/DB/timeout-ошибок. | |

**User's choice:** Широкий retry
**Notes:** —

| Option | Description | Selected |
|--------|-------------|----------|
| Всегда transient | Любая ошибка storage → retry. | |
| Различать по типу | NoSuchKey/404 = terminal; timeout/connection reset = transient. | ✓ |

**User's choice:** Различать по типу

| Option | Description | Selected |
|--------|-------------|----------|
| Transient — повторить всю задачу целиком | Движок отработает заново, идемпотентно перезапишет output. | ✓ |

**User's choice:** Transient — повторить всю задачу целиком

| Option | Description | Selected |
|--------|-------------|----------|
| Terminal | Таймаут почти всегда значит "систематически не укладывается". | |

**User's choice:** Transient, но с ограничением попыток

## Бюджет повторов и backoff для конвертации

| Option | Description | Selected |
|--------|-------------|----------|
| Небольшой (3-5 попыток) | Схоже с webhook (MaxRetry=6), но меньше — временные инфра-сбои обычно решаются быстро. | ✓ |
| Больший (asynq default, ~25) | Максимальная устойчивость, но клиент дольше ждёт ответа. | |

**User's choice:** Небольшой (3-5 попыток)

| Option | Description | Selected |
|--------|-------------|----------|
| Быстрый график в секундах (2с→5с→15с...) | Соответствует характеру проблем: сеть/S3/DB блипы обычно проходят за секунды. | ✓ |
| Постоянная пауза (например 5с между всеми попытками) | Проще, без экспоненциальной кривой. | |

**User's choice:** Быстрый график в секундах
**Notes:** Найдено при разведке кода: `RetryDelayFunc` в asynq общий на сервер, image-очередь сейчас случайно наследует webhook-график (30с→15мин) — нужно исправить.

| Option | Description | Selected |
|--------|-------------|----------|
| Общий бюджет (единая политика) | Таймаут — один из многих transient-случаев, нет нужды в отдельной логике. | ✓ |
| Свой, более строгий лимит (например +1 попытка) | Повторный таймаут на том же файле вряд ли поможет. | |

**User's choice:** Общий бюджет

## Пороги зависания для reconciler'а

| Option | Description | Selected |
|--------|-------------|----------|
| Короткий (1-2 минуты) | Быстрое восстановление при потере enqueue. | ✓ |
| Более длинный (10-15 минут) | Запас на кратковременные пики нагрузки очереди. | |

**User's choice:** Короткий (1-2 минуты)

| Option | Description | Selected |
|--------|-------------|----------|
| Чуть больше таймаута (≈5 минут) | Запас на сам таймаут + I/O + перезапуск. | ✓ |
| Значительно больше (15-30 минут) | Консервативнее, меньше ложных срабатываний. | |

**User's choice:** Чуть больше таймаута (≈5 минут)

| Option | Description | Selected |
|--------|-------------|----------|
| Часто (каждую 1 минуту) | Быстрее реагирует на зависшие задачи. | ✓ |
| Реже (каждые 5 минут) | Меньше нагрузки на БД. | |

**User's choice:** Часто (каждую 1 минуту)

| Option | Description | Selected |
|--------|-------------|----------|
| Все сразу (батчем) | Обычная параллельность воркера (Concurrency) сама сгладит нагрузку. | ✓ |
| С ограничением за проход | Избежать резкого всплеска нагрузки после простоя. | |

**User's choice:** Все сразу (батчем)
