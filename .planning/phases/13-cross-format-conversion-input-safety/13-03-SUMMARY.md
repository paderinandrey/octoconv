# Phase 13 Plan 03: Live E2E Verification of Cross-Format Conversion + OLE-CFB Rejection Summary

**Extended `internal/e2e` with a 6-pair cross-format table (SniffContainer-verified) and a 2-fixture OLE-CFB-rejection table, then ran the full suite live against a freshly built docker-compose stack -- everything passed on the first run, including the existing 6 `->pdf` pairs and the webhook path, confirming CONV-01, CONV-02, and SAFE-01 end to end.**

## Performance

- **Duration:** ~35 min (includes two Docker Desktop disk-exhaustion recoveries and two full image rebuilds)
- **Completed:** 2026-07-10
- **Tasks:** 2 of 3 (Task 3 is the human-verify checkpoint, intentionally not self-approved)
- **Files modified:** 3 (1 modified, 2 created)

## Accomplishments

- `TestCrossFormatConversionE2E`: live table over all 6 intra-family cross pairs (docx→odt, odt→docx, xlsx→ods, ods→xlsx, pptx→odp, odp→pptx), each driven through the real `POST /v1/jobs` → poll → presigned-download flow and validated with a new `assertDownloadIsFormat` helper that calls `convert.SniffContainer` on the downloaded bytes -- the same structural sniff that guards upload input, proving input/output symmetry at the E2E layer (D-03's guarantee, now live-confirmed).
- `TestOLECFBRejectionE2E`: live table over two genuine fixtures (`legacy.doc`, `encrypted.docx`), each asserting a `422 Unprocessable Entity` via a new `postJobExpectStatus` helper (shares a `buildJobRequest` builder with `postJob`, keeping `postJob`'s existing hard-202 contract untouched) and a loose substring check that the body names the remedy ("password").
- Two real CFB fixtures committed: `legacy.doc` produced by running the actual `document-worker` Docker image's `soffice --headless --convert-to doc:"MS Word 97"` against `sample.odt`; `encrypted.docx` produced by encrypting `sample.docx` with `msoffcrypto-tool` (a dev-only fixture-generation tool, no `go.mod` change) and round-trip-verified by decrypting it back to a valid ZIP/docx before committing. Both begin with the `D0 CF 11 E0 A1 B1 1A E1` OLE-CFB magic.
- Full live run against a freshly built stack (`docker compose -f docker-compose.yml -f docker-compose.e2e.yml up -d --build`): all 6 `->pdf` pairs, all 6 cross pairs, and both CFB 422 cases passed -- **zero LibreOffice filter-name corrections were needed**, confirming 13-01's researched filter table is correct against live LO 7.4 (bookworm).

## Decisions Made

- Fixture generation used the actual conversion engine (soffice) and a real crypto tool (msoffcrypto-tool) rather than hand-crafted bytes, so both fixtures are unambiguously genuine per D-07's requirement ("REAL legacy .doc AND a REAL password-protected .docx").
- `postJobExpectStatus` deliberately shares `buildJobRequest` with `postJob` instead of duplicating multipart-building logic, so the two helpers can never silently diverge in request shape.
- The 422 body assertion in `TestOLECFBRejectionE2E` is a loose case-insensitive substring check for "password" (not the full D-06 string), per the plan's explicit guidance to avoid a reworded-message brittle-fail; the 422 status code is the load-bearing assertion.
