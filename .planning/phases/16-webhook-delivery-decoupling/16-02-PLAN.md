<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client webhook receiver ← webhook-worker | webhook-worker dials a client-supplied callback_url (SSRF surface) and signs the payload with the HMAC secret |
| webhook-worker → S3/MinIO | presigns a download_url for done-job payloads |
| operator env → webhook-worker startup | WEBHOOK_SIGNING_SECRET must be present or the signer must not start |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-04 | Spoofing / Information Disclosure (unsigned or missing-secret delivery) | cmd/webhook-worker startup | mitigate | Fail closed: `log.Fatalf("WEBHOOK_SIGNING_SECRET must be set")` when unset (mirrors cmd/worker). Verify gate asserts the string is present. As the ONLY signer now, this binary must never start without it. |
| T-16-05 | Information Disclosure (SSRF via redirect at delivery) | internal/webhook Deliverer (reused verbatim) | mitigate | The consumer-side SSRF defense — the Deliverer's CheckRedirect refusing to follow any redirect (internal/webhook/deliver.go:27-34) — is preserved because internal/webhook is reused verbatim (D-06). The creation-time SSRF guard (validateCallbackURL / WEBHOOK_ALLOW_PRIVATE_IPS) stays in the API process and is unaffected by relocating the consumer. Note: WEBHOOK_ALLOW_PRIVATE_IPS is an API-side var (not read by the deliverer), so it is intentionally NOT wired into webhook-worker; the consumer's guard is redirect-refusal, not a re-check. |
| T-16-06 | Information Disclosure (secret in logs) | cmd/webhook-worker logging | mitigate | Signing secret is read into a []byte and never logged; only lifecycle strings are logged. Code review + no `log.*signingSecret` reference. |
| T-16-07 | Tampering (image-worker silently keeps webhook role) | cmd/worker/main.go | mitigate | Clean cut (D-03): TypeWebhookDeliver + sweeper removed from cmd/worker; verify gate asserts no TypeWebhookDeliver/reconciler/NewSweeper references remain, so exactly one binary delivers webhooks. |
| T-16-SC | Tampering | npm/pip/cargo installs | accept | No package-manager install; Go build of existing modules only. Dockerfile installs only ca-certificates from Debian's signed apt repos. No legitimacy audit required. |
</threat_model>
