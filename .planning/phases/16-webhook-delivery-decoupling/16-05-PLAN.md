<objective>
Close the two verified Phase 16 reliability gaps (16-VERIFICATION.md, status: gaps_found) that were carried over from the code review (16-REVIEW.md CR-01 and WR-01). Both share one root cause: the `PGAdvisoryLock`'s dedicated Postgres connection has a write-only lifecycle — it is acquired but never properly returned under any code path.

1. CR-01 (Critical): `PGAdvisoryLock.TryAcquire`'s error path hard-closes the underlying pgconn but never `Release()`s the `*pgxpool.Conn` wrapper, permanently burning one pool slot per transient Postgres fault → eventual crash-free pool exhaustion → silent halt of webhook delivery and sweeping (reintroduces the exact silent-loss failure mode Phase 16 exists to eliminate, via a different mechanism).
2. WR-01 (Warning, live-reproduced): `PGAdvisoryLock` has no `Close()`/`Release()` method, so the dedicated advisory-lock connection is never returned at shutdown; `defer pool.Close()` in `cmd/webhook-worker/main.go` blocks forever on SIGTERM (container logged "bye 👋" but stayed Up 20+s until SIGKILL).

Adding Close() introduces a shutdown-time concurrency interleaving — the deferred `lock.Close()` on the main goroutine can run while the `go sweeper.RunWithLock` goroutine is still mid-flight inside `TryAcquire`, both touching `l.conn` with no synchronization. A `sync.Mutex` on PGAdvisoryLock serializes all `l.conn` access across the two goroutines, making every interleaving race-free.

Purpose: Restore the "надёжно" (reliably) clause of the project's Core Value for the webhook-delivery path — no silent stop on transient DB faults, and clean bounded-time graceful shutdown on every redeploy.
Output: The two source fixes (plus the mutex) and a DATABASE_URL-guarded regression test asserting all three behaviors.
</objective>

- `func (c *Conn) Conn() *pgx.Conn`        — the underlying raw connection (has Close(ctx))

                                             (safe to call multiple times — subsequent calls are no-ops).

Current PGAdvisoryLock (internal/reconciler/reconciler.go:108-156):
- struct fields: `pool *pgxpool.Pool`, `conn *pgxpool.Conn`
  (this fix ADDS a `mu sync.Mutex` field guarding conn lifecycle across goroutines).
- `NewPGAdvisoryLock(ctx, pool) (*PGAdvisoryLock, error)` — acquires ONE dedicated conn, holds it.
- `TryAcquire(ctx) (bool, error)` — lazily re-acquires if l.conn==nil, then
  `SELECT pg_try_advisory_lock($1)`; on query error it does:
