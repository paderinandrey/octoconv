## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub runner → third-party actions | Each `uses:` pulls external action code into the runner (supply chain) |
| Committed dev creds → CI job logs | docker-compose literals (API_KEY_SALT, WEBHOOK_SIGNING_SECRET, DB pass) are visible to every CI step |
| e2e SSRF-guard relaxation → non-e2e jobs | docker-compose.e2e.yml opts out of the webhook SSRF guard for the test receiver |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-01 | Tampering | third-party action versions (`uses:`) | mitigate | Pin every action to its trusted major tag (actions/checkout@v7, actions/setup-go@v6, docker/setup-buildx-action@v4, docker/bake-action@v7, actions/upload-artifact@v7); use ONLY actions/* and docker/* — no unvetted marketplace actions (e.g. no jlumbroso/free-disk-space; disk cleanup is a plain shell step instead). |
| T-19-02 | Information disclosure | committed dev creds echoed to CI logs | accept | Values are explicit dev-only placeholders ("dev-only-change-me-in-real-deploys"), not real secrets, for an internal service never deployed from CI. Do not add steps that dump full `env`/compose config; reference no `secrets.*`. |
| T-19-03 | Elevation of privilege / Info disclosure | docker-compose.e2e.yml SSRF opt-out escaping test context | mitigate | `.e2e.yml` is referenced ONLY in the e2e job's `docker compose -f ... -f docker-compose.e2e.yml` invocations; no other job (gate/race/docker-build) references it. Verified by acceptance criterion in Task 1. |
| T-19-SC | Tampering | package-manager installs (npm/pip/cargo) | accept (n/a) | This phase performs NO package-manager installs — it is GitHub Actions YAML plus the Go toolchain provisioned by setup-go from go.mod. No Package Legitimacy Gate applies. |
