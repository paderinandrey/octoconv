<ms_cfb_facts>
The executor MUST NOT re-research the format. Everything needed is below. Where any offset is uncertain, the FIXTURE-DRIVEN acceptance is the source of truth: correct classification of the real legacy.doc/encrypted.docx fixtures is the pass condition (Task 1 verify), so a wrong constant fails a unit test rather than shipping.

Header (the first 512 bytes, ALWAYS 512 regardless of sector size), little-endian:
  - offset 0:  8-byte signature D0 CF 11 E0 A1 B1 1A E1 (already guaranteed by the caller's IsOLECFB, but re-check defensively)
  - offset 30: SectorShift  uint16 — sector size = 1 << SectorShift. Valid: 0x0009 (512-byte, major v3) or 0x000C (4096-byte, major v4). ANY other value → CFBUnknown.
  - offset 44: NumFATSectors uint32
  - offset 48: FirstDirSectorLocation uint32 — first sector of the directory stream.
  - offset 68: FirstDIFATSectorLocation uint32
  - offset 72: NumDIFATSectors uint32
  - offset 76: DIFAT array — the first 109 entries, each a uint32 sector number, filling bytes 76..512. These point at the FAT sectors.

Sector addressing: the byte offset of sector number N in the file = (N + 1) * sectorSize (sector 0 begins at offset sectorSize, NOT 0; for v4 the 512-byte header is followed by padding up to the first sector). Bounds-check (N+1)*sectorSize + sectorSize <= size before EVERY sector read.

FAT special sentinels (uint32): 0xFFFFFFFE = ENDOFCHAIN, 0xFFFFFFFF = FREESECT, 0xFFFFFFFD = FATSECT, 0xFFFFFFFC = DIFSECT. Treat anything >= 0xFFFFFFFA as a terminator/invalid (stop the chain).

FAT build: read each FAT sector named by the header DIFAT (first 109 entries). Each FAT sector holds sectorSize/4 uint32 next-pointers; concatenated they form FAT[k] = sector following sector k. This parser supports ONLY the 109 header-DIFAT entries: if FirstDIFATSectorLocation is not ENDOFCHAIN (i.e. the file declares additional DIFAT sectors, meaning a >~7 MB file), return CFBUnknown fail-closed — such files are outside this parser's bound and are still rejected downstream, just with the generic message (this is safe per D-04/Pitfall 11; both target fixtures are ~9 KB and never need DIFAT sectors). This DIFAT-bail is Claude's discretion to instead follow bounded, but the simple bail is recommended.

Directory walk: start at FirstDirSectorLocation and follow the FAT chain of directory sectors (visited-set guarded). Each directory sector holds sectorSize/128 directory entries (each entry 128 bytes). Read EVERY allocated entry linearly — do NOT walk the red-black sibling/child tree; a flat scan of all directory entries enumerates every stream name and avoids the tree-traversal bug surface (Pitfall 10). Per 128-byte entry, little-endian:
  - offset 66: ObjectType uint8 — 0x00 unallocated (skip), 0x01 storage, 0x02 stream, 0x05 root storage.
  - offset 64: NameLength uint16 — length in BYTES of the name field including the trailing UTF-16 NUL; valid range 0..64. If >64 or odd-in-a-way-that-overflows, treat the entry as invalid and skip it (do NOT abort the whole parse — a single bad entry is not a cycle).
  - offset 0:  Name, 64 bytes, UTF-16LE. Decode the first NameLength bytes, then strip the trailing NUL code unit. (Character count = NameLength/2 - 1.)

Classification (D-03), over the SET of decoded entry names:
  - contains "EncryptionInfo" OR "EncryptedPackage"  → CFBEncrypted  (encrypted WINS if legacy markers also present)
  - else contains "WordDocument" OR "Workbook" OR "Book" OR "PowerPoint Document" → CFBLegacy
  - else → CFBUnknown
  (Note the Book vs Workbook note: include BOTH in the legacy set; the .xls-family variant is empirically confirmed by the fixture-driven test rather than assumed.)

DoS hardening (D-04, MANDATORY — this is CFB-02's whole point):
  - Maintain a visited-set (map[uint32]bool) of sector indices for EVERY FAT-chain follow (both the directory-sector chain and any chain walk). A repeat visit → return CFBUnknown immediately (cycle rejected, never loop).
  - Hard cap: walk at most 4096 directory sectors; if the chain would exceed it → CFBUnknown.
  - Every read is bounds-checked against `size` before ReadAt; a short read or an out-of-range declared offset/count → CFBUnknown.
  - NEVER panic, NEVER loop unbounded, NEVER read stream CONTENT, NO mini-FAT, NO decompression (D-02).
