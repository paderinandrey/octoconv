## D-05 REFINEMENT — DROP the migrate hook Job (BLOCKER fix, verified)
The original plan added a post-install migrate hook Job (running cmd/migrate) AND left Dockerfile.api building the migrate binary. That design is REMOVED. Reason (verified against source):
  - `cmd/api/main.go` (lines 49-51) already calls `db.Migrate(ctx, pool)` UNCONDITIONALLY on every startup. This is the proven migration mechanism — docker-compose has relied on it since v1.0.
  - `internal/db/db.go` `Migrate` has NO locking (plain schema_migrations existence-check + per-file tx apply). A migrate hook Job and cmd/api both unblock on the same postgres-readiness instant → two concurrent unlocked `db.Migrate` runs can collide on the same migration; the hook's `backoffLimit: 0` then hard-fails `helm install --wait` (breaks SC1/SC2).
  - `grep -rn "db.Migrate" cmd/` confirms ONLY cmd/api and cmd/migrate call it. The four workers (worker, document-worker, chromium-worker, webhook-worker) do NOT self-migrate: their startup is `db.Connect` (a ping only) → asynq server on Redis; they touch Postgres only when handling a job, which cannot happen until api has accepted it (and api migrates before serving). So workers tolerate schema-not-ready by construction — no schema query fires until after api has landed it. (webhook-worker's reconciler sweep loop may error on its first sweeps until api migrates, then succeed under the advisory lock — benign, self-healing.)
Resolution: the api Deployment's single replica self-migrates on startup; migration ordering is satisfied by api's own startup + k8s restart loop. Record this as the K8S-02 D-05 refinement in the SUMMARY: "migrate via api self-migration (existing behavior); createbucket via post-install hook". Dockerfile.api is left UNCHANGED (it never needed the migrate binary — that build line existed solely for the now-dropped hook).

## createbucket hook phase (surfaced, not silent)
createbucket runs as `post-install,post-upgrade` (hook-weight "0"), with an initContainer that waits for the `minio` Service to be Ready. Post-* (not pre-*) because the `minio` Service is a main-release resource that does not exist during the pre-install phase — a pre-install Job (or its wait-for-minio initContainer) could never reach MinIO. Stateful deps stay normal main resources (never hook-deleted), keeping upgrades data-safe.
