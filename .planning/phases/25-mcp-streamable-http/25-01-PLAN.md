<objective>
Build the Go side of the streamable-HTTP MCP endpoint: refactor `internal/mcpserver` so a `*mcp.Server` can be constructed per HTTP request with the caller's own API key (per-request pass-through, D-03) and with a result mode that returns presigned-only results for remote callers (D-04), then add the `cmd/mcp-http` binary with a 401 net/http middleware wrapping the go-sdk `NewStreamableHTTPHandler`. The stdio binary (`cmd/mcp-server`) and the existing `NewServer(cfg, client)` API must stay behaviorally identical. Because the go-sdk reuses a session-bound server WITHOUT re-invoking getServer when an `Mcp-Session-Id` matches an existing stateful session (streamable.go:400), the binary also installs a session→key binding middleware that rejects any request whose key does not match the session's creator — closing a cross-key hijack that would otherwise let K2 execute as K1 in the stateful fallback.

The first task is a cheap live spike gate: prove go-sdk v1.6.1's `Stateless: true` still delivers `convert_file`'s in-flight progress notifications to an HTTP client (D-02, flagged LOW confidence by research) before the binary commits to stateless.

Purpose: Serves MCPH-01 (streamable-HTTP endpoint, per-request caller-key pass-through, same 5 tools) and MCPH-02 (remote-usable convert_file result via the presigned-only contract).
Output: `cmd/mcp-http` binary, ResultMode-aware `internal/mcpserver`, exported `auth.ParseAPIKey`, and unit + spike tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → mcp-http | Untrusted `Authorization: ApiKey <key>` header crosses here; the header is the only credential |
| mcp-http → OctoConv API | Per-request forwarded caller key; the API (not mcp-http) does the hash-compare against `clients` |
| mcp-http → presigned S3 URL | Self-authorizing V4 URL returned to the caller (remote mode) |
