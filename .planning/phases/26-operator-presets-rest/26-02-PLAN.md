<objective>
Close the single blocking gap from 26-VERIFICATION.md (CR-01 in 26-REVIEW.md): deactivate-then-recreate of the same preset name always fails with a raw 500, permanently "bricking" that name via REST.

Root cause (pre-existing Phase 18/20 debt, newly reachable through Phase 26's operator REST surface): `internal/presets/repo.go` `Create` hardcodes `const version = 1` and its duplicate pre-check excludes all but ACTIVE rows, while the DB unique indexes `presets_system_uq (name, version) WHERE scope='system'` and `presets_user_uq (client_id, name, version) WHERE scope='user'` (0001_init.sql:35-36) cover INACTIVE rows too. A second Create for a previously-deactivated name collides on the unique index; the pgx unique-violation is never mapped to `presets.ErrAlreadyExists`, so it surfaces as an opaque 500 with no recovery path (`Update` requires an active row).

Purpose: make the "deactivate a bad preset, recreate it later" workflow — implied by the no-hard-delete design — actually work over REST, satisfying OPER-01's "manageable via REST" clause for both scopes.
Output: a version-bumping, race-safe `Create` in `internal/presets/repo.go` plus a DB-backed regression test in `internal/presets/repo_test.go`.

This fix belongs entirely in `internal/presets` (repo.go + tests). It does NOT touch the API layer, the `PresetAdmin` interface signature, or any migration. It fixes system AND user scope generically (single shared code path). It does NOT address WR-01/WR-02/WR-03/IN-01/IN-02 — those are non-blocking and tracked separately.
</objective>
