## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| in-namespace pod → api :9090/metrics | Metrics NetworkPolicy governs who may scrape the queue-depth signal |
| Prometheus HTTP API :9090 → in-namespace callers | Prometheus stores/serves queue-depth; its query API is reachable within the octoconv namespace |
| KEDA operator (keda ns) → Prometheus :9090 (octoconv ns) | Cross-namespace scaler query path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-04 | Information Disclosure | api :9090 metrics port newly exposed on the Service | mitigate | networkpolicy-metrics.yaml (fixed this plan) admits :9090 only from the component=prometheus pod in the octoconv namespace; all other pods are default-denied once the Ingress policy selects api's tier=app pod. (Environmentally unenforced on OrbStack — no CNI policy controller — but correct for any conformant CNI; verified via render, per Phase 24 precedent.) |
| T-27-05 | Information Disclosure | Prometheus HTTP API :9090 queryable in-namespace | accept | Prometheus holds only operational queue-depth/job-outcome counters (no PII, no secrets); it ships behind `prometheus.enabled` (off by default) and has its own component label distinct from tier=app. Locking down Prometheus's own query API to only the KEDA namespace is a future hardening item, disproportionate for this internal dev cluster. |
| T-27-06 | Elevation of Privilege / Tampering | ScaledObject can drive Deployment replica counts | mitigate | ScaledObjects are chart-owned, gated (keda.enabled AND prometheus.enabled), and bounded: minReplicaCount 0 / maxReplicaCount ≤4 per class (D-06, sized for the OrbStack VM). webhook-worker is hard-excluded (no ScaledObject) so the sweeper singleton can never be scaled to zero (Pitfall 2). A compromised metric could at most scale a class to its ceiling, not unbounded. |
| T-27-07 | Denial of Service | Prometheus unreachable → workers can't scale up | mitigate | `fallback: { failureThreshold: 3, replicas: 1 }` on every ScaledObject (D-07) holds 1 worker per scaled class and keeps processing if Prometheus queries fail — fail-safe toward availability. `ignoreNullValues: true` prevents routine api restarts from spuriously tripping fallback (Pitfall 4). The `and` co-gate also removes the misconfiguration path where a ScaledObject exists but its Prometheus target does not. |
| T-27-SC | Tampering (supply chain) | KEDA Helm chart + prom/prometheus image (external artifacts) | accept | Both verified against authoritative sources in RESEARCH.md Package Legitimacy Audit: KEDA chart `keda` v2.20.1 (official kedacore Helm repo + GitHub Releases API, live-fetched) and `prom/prometheus:v3.13.1` (official `prom/` Docker Hub org + Prometheus GitHub Releases API). Both pinned to concrete non-`latest` versions in values.yaml / plan 03. No npm/pip/cargo installs — slopcheck not applicable. KEDA install itself is a plan-03 gate step. |
