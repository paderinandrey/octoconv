## Форма таймстамп-доказательства

| Option | Description | Selected |
|--------|-------------|----------|
| CSV-семплер + PNG-график | Скрипт каждые ~5s пишет (ts, queue_depth, ready-реплики) в CSV; после прогона рендер PNG с двумя сериями на одной оси | ✓ |

## Механика SC3: долгий джоб vs даунскейл

| Option | Description | Selected |
|--------|-------------|----------|
| pod-deletion-cost на занятом поде | Низкий cost на занятый под → реальный KEDA-даунскейл 2→1 детерминированно убирает его; SIGTERM-путь штатный | ✓ |
| Ретраить до естественного попадания | Честнее, но ≈50/50 на попытку с долгими джобами — недетерминированное время гейта | |
| kubectl delete pod занятого | Детерминировано, но это НЕ KEDA-даунскейл — SC3 требует «survives a KEDA downscale event» | |

| Option | Description | Selected |
|--------|-------------|----------|
| Сгенерированный тяжёлый docx | Генератор под целевые ~200s, калибруется пробным прогоном; с запасом от timeout 300s | ✓ |
| docx→pdf с pdfa=true | veraPDF p95 всего ~4.65s — не даёт нужной длительности | |
| Снизить планку до ~60-90s | Быстрее, но SC3 буквально говорит «~200s» | |

| Option | Description | Selected |
|--------|-------------|----------|
| Тройная проверка | done+скачивание; ровно один queued→active в job_events; таймстампы SIGTERM-до-завершения и под ушёл до grace | ✓ |
| Только done + одна попытка | Не доказывает, что джоб реально был под SIGTERM во время работы | |
