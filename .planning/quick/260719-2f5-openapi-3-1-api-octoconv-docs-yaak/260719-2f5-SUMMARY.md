## Decisions Made
- `ApiKeyAuth` modeled as `type: apiKey` (per plan instruction) rather than `type: http, scheme: apikey`, since the latter is not a spec-valid securityScheme value and would fail import/lint. The "ApiKey " literal prefix is documented in the scheme's `description` since `apiKey` schemes have no structural way to express a prefix.
- `opts` response fields (`JobCreated.opts`, `JobStatus.opts`) use `anyOf: [AudioOpts, generic object]` instead of `oneOf` — initial `oneOf` attempt failed redocly's example-validation rule because a generic `additionalProperties: true` object schema always structurally matches AudioOpts too, so no value can satisfy "exactly one" schema. `anyOf` accurately reflects that opts are AudioOpts for the audio engine and an arbitrary engine-specific object otherwise.
- Reused `components.responses` (Unauthorized/NotFound/TooManyRequests/InternalError) across operations to avoid duplicating the same `Error`-schema response body 30+ times, keeping the spec both accurate and lint-friendly.

## Deviations from Plan

None - plan executed exactly as written. Two redocly lint warnings were deliberately left unresolved (both are informational-severity, not errors, and "Reasonable spec-lint warnings ... may be accepted if benign" per Task 2's action text):

1. `info-license`: no `license` field. This is an internal, unlicensed service — there is no real license to declare, and fabricating one (e.g. an SPDX identifier or URL) would misrepresent the project.
2. `operation-4xx-response` on `GET /healthz`: the handler (`internal/api/handlers.go:handleHealth`) genuinely only ever returns 200 or 503 — there is no 4xx path for this route, and inventing one would violate the plan's "describe the API AS IT IS in code" objective.
