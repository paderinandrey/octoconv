# Feature Research

**Domain:** Offline audio-transcription-as-a-service (whisper.cpp engine class, v1.7 audio engine)
**Researched:** 2026-07-17
**Confidence:** MEDIUM (whisper.cpp CLI flags/build options verified against upstream README/discussions; exact JSON field names for `-oj`/`-ojf` and CPU real-time-factor numbers are extrapolated from adjacent tooling and should be spot-checked against the actual pinned whisper.cpp version before implementation)

## Framing

This is **not** a generic "add speech-to-text" feature list. It is scoped to OctoConv's existing architecture (async job pipeline, `Converter`/`Registry` abstraction, validated `opts` allowlists, presets, magic-bytes content validation, webhook delivery, MCP tools) and to the SEED-001 lesson-recording vertical: **this milestone ships transcription only** — mistake-analysis and the spaced-repetition deck are a later milestone that will *consume* this milestone's result contract. Every recommendation below is filtered through "does this fit the existing `Converter.Convert(ctx, inPath, outPath, opts)` shape, the existing opts-allowlist pattern, and the existing single-input/single-output-ordinal-0 constraint documented in CLAUDE.md?"

**Central design finding:** whisper.cpp's own output-format flags (`-otxt`, `-osrt`, `-ovtt`, `-oj`/`-ojf`) map naturally onto OctoConv's existing `(source_format, target_format)` `Pair` mechanism — audio's "target format" is not another audio codec, it's the *transcript shape* (`txt`/`srt`/`vtt`/`json`). This means output-format selection needs **zero new API surface**: it's just more `Pair`s in the audio `Converter.Pairs()`, exactly like image's png→webp or document's docx→pdf. `AudioOpts` (language, translate-to-English) stays orthogonal to the pair, mirroring how `DocOpts.PDFProfile` is orthogonal to the docx→pdf pair today.

## Feature Landscape

### Table Stakes (Users Expect These)

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| mp3/wav/m4a input support | These three cover the overwhelming majority of "lesson recording from a phone or Zoom" sources: iPhone Voice Memos and most Android recorders default to `.m4a` (AAC-in-MP4), Zoom local/cloud recordings default to `.m4a`/`.mp4` audio-only exports, and `.mp3`/`.wav` are the universal lowest-common-denominator formats every recording/editing tool can export. Missing any of these three makes the product feel broken for its primary buyer persona. | MEDIUM | whisper.cpp's built-in decoder (miniaudio) natively handles **WAV, MP3, FLAC, OGG/Vorbis only** — it does **not** decode M4A/AAC or Opus without either (a) an FFmpeg pre-normalization step, or (b) building whisper.cpp itself with `WHISPER_COMMON_FFMPEG=yes` (links libavformat/libavcodec into the binary). Recommend (a): shell out to `ffmpeg -i in -ar 16000 -ac 1 -c:a pcm_s16le out.wav` as an explicit first `runCommand` step inside `Convert()`, then run `whisper-cli` on the normalized WAV. This mirrors the project's existing "hardened exec, one external tool per boundary" philosophy better than a nonstandard whisper.cpp build, keeps ffmpeg's very well-tested demuxer/decoder in front of untrusted client bytes, and gives a natural place to run `ffprobe` for a cheap upfront duration/size sanity check (see below). |
| ogg/opus input support | WhatsApp/Telegram voice-note forwards (`.opus`/`.oga`, OGG container + Opus codec) are a realistic secondary source for tutor/student lesson snippets shared outside Zoom. | MEDIUM | Same ffmpeg-normalization path as m4a covers this for free — ffmpeg decodes Opus-in-Ogg natively, whisper.cpp's own miniaudio decoder does not (it only handles Vorbis-in-Ogg). Treat mp3/wav/m4a/ogg/opus as one input-format list all routed through the same ffmpeg-normalize step, not as format-specific code paths. |
| Magic-bytes content validation for audio containers | Matches the project's fail-closed pattern already enforced for images (PNG/JPEG/WebP/HEIC/TIFF) and OLE-CFB documents — extension-trust is explicitly rejected elsewhere in this codebase (`NormalizeFormat`/`Sniff` operate on detected bytes, not the client-supplied extension). Skipping this for audio would be an inconsistent security posture for the exact same threat model. | MEDIUM-HIGH | WAV (`RIFF`...`WAVE`) and OGG (`OggS`) have hard, unambiguous magic-byte signatures — LOW effort, mirrors PNG/TIFF today. M4A is ISOBMFF (`ftyp` box + brand, e.g. `M4A `, `M4B `, `isom`, `mp42`) — directly reuses the HEIC-brand-table pattern already in `internal/convert/sniff.go`. **MP3 is the hard case**: files with an ID3v2 tag have a clean `ID3` magic prefix, but bare MP3 (no ID3 tag) has no fixed signature — only a frame-sync bit pattern (`0xFFEx`/`0xFFFx`) that is a heuristic, not a deterministic match, and can false-positive/negative. Flag this explicitly for PITFALLS research/phase design: the project's "detected format is the single source of truth" invariant is harder to uphold cleanly for MP3 than for any format registered so far. |
| Segment-level timestamps in every transcript | Any transcription-as-a-service that doesn't let a user (or a downstream consumer) jump to *where* in the audio a piece of text came from feels incomplete — this is the baseline expectation set by every mainstream STT product (Whisper API, Otter, Zoom's own transcript, YouTube captions). | LOW | Native to whisper.cpp: `-osrt`/`-ovtt` produce timed-text directly; `-oj` includes per-segment `start`/`end`/`text`. No extra engineering beyond wiring the flag through `Convert()`. |
| Auto language detection by default | Lesson recordings will span many student-language pairs; forcing every client to know and declare the spoken language up front (especially the *student's* language, which the calling internal service may not always know) is a needless barrier for an internal API whose whole value proposition is "just throw the file at it." | LOW | whisper.cpp supports `-l auto`, but its **own CLI default is `-l en`, not auto** — this must be an explicit, deliberate default choice in `AudioOpts` (default `"auto"` unless the client sets `language`), not an assumption that whisper.cpp's own default is safe to rely on. |
| Duration/size guard before the expensive step | Lesson recordings run up to ~1h; without an upfront duration cap a client (or an attacker) can submit a wildly long or malformed file and tie up worker capacity for a very long, unbounded time — this is the same "resource-exhaustion via crafted input" class of problem the project already solved for images (decompression-bomb pixel-count guard, `MAX_IMAGE_PIXELS`) and accepted as residual risk for documents (`DOCUMENT_ENGINE_TIMEOUT` only, no complexity guard). | MEDIUM | Run `ffprobe` (bundled with ffmpeg) to read the declared duration *before* invoking whisper-cli, and reject (422, before S3 write, matching the existing magic-bytes-before-storage ordering principle) anything over a configurable ceiling (e.g. `AUDIO_MAX_DURATION_SECONDS`, default comfortably above 60 min to allow real lesson recordings plus buffer). This is strictly upfront/declared-duration validation, analogous to but weaker than the image pixel-count guard — a crafted file could still lie about its duration; treat unbounded worker-side wall-clock protection as the job of `AUDIO_ENGINE_TIMEOUT` (see dependencies below), not this check alone. |
| Async job pipeline (upload → queue → worker → S3 result + presigned URL + webhook) | Already built, engine-agnostic by design. | ZERO (inherited) | No new work — this is the entire point of the `Converter`/`Registry` abstraction. Listed here only to make explicit that it is *not* a gap; audio "just" needs to be a new `Converter` implementation plus a new engine-class queue/worker/KEDA wiring, following the exact document/chromium precedent. |
| Own asynq queue + worker binary + retry semantics + KEDA ScaledObject | Matches the established engine-class-isolation pattern (`document`, `html` each got their own queue, worker binary, container, and `ScaledObject`); a shared queue would let a long audio job starve fast image/document jobs behind it. | MEDIUM | Structurally identical to the document-engine precedent (Phase 10: `TypeAudioConvert`/`QueueAudio`, `cmd/audio-worker`, `Dockerfile.audio-worker`, derived unique-lock TTL, reconciler engine-routing entry). The one real decision is transient-vs-terminal classification for `AUDIO_ENGINE_TIMEOUT`: unlike the image engine (timeout = transient, cheap to retry) the document engine already treats timeout as **terminal** because retrying an inherently-slow job just wastes another full timeout window for no better odds of success. Audio should almost certainly follow the **document precedent (terminal)**, not the image one — a 1h recording that times out once will time out again with the same inputs. |

### Differentiators (Competitive Advantage)

| Feature | Value Proposition | Complexity | Notes |
|---------|--------------------|------------|-------|
| `target_format=json` as the richest, SEED-001-forward-compatible output contract | Directly answers the "what serves both humans and downstream ML" question: request `json` and you get per-segment `start`/`end`/`text` (and, with word-level timestamps enabled, per-word timing) — exactly the structured, timestamp-aligned data a future mistake-analysis/spaced-repetition pass needs to point back at *which second of audio* a mistake occurred, without OctoConv having to guess at or design a bespoke schema now. `txt`/`srt`/`vtt` remain available as the same-engine, human-consumption targets (subtitle players, quick review) via the identical `(source, target)` pair mechanism — no separate "output format" opt needed, no bespoke API. | MEDIUM | Requires whisper.cpp's `-oj`/`-ojf` plus word-level timestamp flags (`-ml 1` word-splitting or equivalent) enabled for the `json` target specifically. **Verify exact field names against the pinned whisper.cpp version before building the mistake-analysis milestone against them** — the JSON schema is not stabilized/versioned upstream the way SRT/VTT are, and `-ojf` ("full") vs `-oj` differ in exactly which per-token/per-word fields are present. This is the single most important forward-compatibility decision for SEED-001 — get the `target=json` contract right now even though nothing consumes the extra fields yet, because the mistake-analysis milestone should not need a `Converter` contract change to get timestamp-aligned text. |
| Explicit `language` + `translate` opts via the existing validated-opts mechanism | Bilingual tutor/student sessions (tutor speaks English, student speaks their target language, possibly mid-utterance code-switching) are exactly where auto-detect is least reliable — Whisper's language-detection runs once on the first ~30s of audio and applies that single language for the whole file, so a session that opens in English small-talk before switching to the target language can get auto-detect wrong for the bulk of the recording. Letting the calling internal service pass a known/expected primary language when available is a real accuracy lever, and `translate` (spoken-language → English text) is a genuinely differentiating capability for a tutor-facing internal service reviewing recordings across many languages. | LOW-MEDIUM | Follows the exact `DocOpts`/`HTMLOpts` pattern already in `internal/convert/opts.go`: a closed `AudioOpts` struct (`Language string`, `Translate bool`), strict-parsed via the shared `checkStrictObject` helper, `DisallowUnknownFields`, validated against a closed language-code allow-list (or accept any well-formed 2-letter code and let whisper.cpp itself reject unsupported ones — decide during phase planning), an `AudioOptsFromMap` round-trip, and an `ValidateAudioApplicability` gate scoped to `EngineAudio`. **Honest limitation to carry forward, not solve here**: whisper (upstream and whisper.cpp) has no true multi-language/code-switching mode — `language` selects one language for the whole file, and `translate` only ever targets English (not arbitrary language pairs). A single job cannot cleanly transcribe "tutor in English, student in Spanish" as two correctly-tagged languages in one pass; that is a real product gap worth flagging to the roadmap, not a bug to fix inside this opts mechanism. |
| Named presets for common transcription configs | Reuses the existing preset mechanism (`internal/presets`, `POST /v1/jobs preset=<name>`) with zero new plumbing beyond wiring `AudioOpts` into `internal/presets/optscheck.go`'s allowlist, exactly as `DocOpts`/`HTMLOpts` are wired there today. Concretely valuable for the buyer persona: a language-school client can define once and reuse presets like "transcribe, auto-detect, json output" or "transcribe-and-translate-to-english, srt output" instead of re-specifying opts on every job. | LOW | Mechanically identical to existing preset wiring; the only new work is the `AudioOpts` allowlist entry itself (already counted above). |
| MCP tool exposure (`convert_file`, `list_supported_formats`) | `list_supported_formats` is registry-derived (`Registry.Classes()`) and `convert_file` is engine-agnostic — both pick up the new audio `Pair`s automatically once `AudioConverter` registers into `convert.Default`, the same way html/document pairs required no MCP code changes. | ZERO (inherited) | Confirms the registry-driven MCP design is paying off exactly as intended; call this out as a positive validation of the existing architecture rather than new work. |
| Finer-grained job progress for long transcription jobs | Existing MCP `convert_file` progress is coarse — it polls job status per tick (queued/active/done), which is adequate for near-instant image/document conversions but is a weak signal for a job that may legitimately run tens of minutes on CPU. whisper-cli exposes a `--print-progress` flag that streams a live percentage to stderr during decode. | HIGH (defer) | The existing `runCommand` helper (`internal/convert/exec.go`) buffers stdout/stderr entirely and returns them only after the process exits — it does not stream. Tapping `--print-progress` for real percentage-based progress would require a genuinely new exec primitive (a streaming-stderr-parser variant of `runCommand`) shared with nothing else in the codebase today. Recommend treating this as an explicit **future differentiator, not v1.7 scope** — ship with the existing poll-based "active" status first, and revisit only if long-job UX actually becomes a complaint. |

### Anti-Features (Commonly Requested, Often Problematic)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|------------------|-------------|
| Real speaker diarization (labeled "Tutor:" / "Student:" turns) | Obviously valuable for a tutor/student lesson recording — the SEED-001 mistake-analysis milestone would clearly benefit from knowing *who* said the mistaken phrase. | whisper.cpp ships two diarization mechanisms, and **neither is production-grade for this use case**: (1) `-tdrz`/tinydiarize is explicitly described upstream as an experimental prototype, finetuned on a *single* English-only model (`small.en-tdrz`) — it marks speaker-turn boundaries with a special token, not speaker identity, and has no non-English model at all, which rules it out for exactly the bilingual tutor/student recordings this milestone is framed around. (2) `-di`/`--diarize` ("stereo audio diarization") requires the two speakers to already be on separate stereo channels, which single-track phone/Zoom-mixed-audio recordings are not. Real diarization (e.g. pyannote-style embeddings) is a Python/PyTorch-heavy pipeline that conflicts with the locked engine decision (whisper.cpp, offline, single hardened CLI binary) and the project's "no new heavy runtime dependency per engine container" pattern. | Ship the `json` transcript contract now with the speaker concept simply absent (no `speaker` field), so it can be added later as an additive field without breaking existing consumers, rather than bolting on an unreliable/English-only diarization signal now. If real diarization is ever justified, treat it as its own future milestone/engine decision, not a bolt-on opt on the whisper.cpp audio engine. |
| Real-time/streaming transcription | "Live" transcription feels like a natural extension once transcription exists at all, and some tutoring products market live captions during the session itself. | Fundamentally incompatible with the existing transport: lesson recordings are uploaded post-hoc as a whole file (multipart `POST /v1/jobs` → queue → worker → S3 → webhook), not a live media stream. Streaming would require an entirely new transport (WebSocket/gRPC/chunked-upload) foreign to every other engine class and to SEED-001's actual workflow (tutor uploads a finished recording after the lesson, not during it). | None needed for this milestone's actual use case — SEED-001 is explicitly post-lesson analysis, not live captioning. |
| Arbitrary target-language translation (not just to English) | "Translate the lesson to French for a French-speaking reviewer" sounds like a small extension of the `translate` opt. | whisper(.cpp)'s built-in `--translate` is hardcoded to translate-**to-English only** — there is no other target language in the model itself. Supporting arbitrary target languages would require bolting on a wholly separate machine-translation engine/model, which is a different engine-class decision entirely, not an opts tweak on the whisper.cpp converter. | If arbitrary-language translation is ever needed, scope it as its own future engine class/milestone (a real MT model), not as an `AudioOpts` field that quietly does something whisper.cpp cannot actually do. |
| Mistake detection / grammar analysis / spaced-repetition deck generation in this milestone | This is literally SEED-001's stated end-value ("a list of the student's top mistakes ... a ready-made spaced-repetition deck"), so it's tempting to reach for it while already in the audio-engine milestone. | Explicitly out of scope per `.planning/PROJECT.md` ("SEED-001 фундамент: контракт результата транскрипции ... следующий милстоун"/"анализ ошибок отложен") — this milestone's job is the transcript *contract*, not the analysis that consumes it. Building analysis now would couple an LLM/NLP pipeline decision to an engine-class-plumbing milestone and blow the scope. | Get the `json` transcript contract (segment + word timestamps, language, confidence where available) right *now*, and let the next milestone build mistake-analysis as a consumer of stored transcript JSON — no engine/result-contract changes needed later if this contract is designed with that consumer in mind from day one. |
| Multi-file / batch-in-one-job transcription (e.g., upload a whole course's worth of recordings as one job) | Feels efficient for a language-school client with many lesson recordings to process at once. | Contradicts the documented architectural constraint that the entire pipeline (API validation, worker read/write paths) currently only reads/writes ordinal `0` of `job_inputs`/`job_outputs`, even though the schema itself supports multiple ordinals. Introducing batch-audio would be the first place in the codebase to actually need multi-input/output handling — a cross-cutting change well beyond a single new engine class. | One job per recording, same as every other engine class today; a client wanting batch throughput submits N jobs (the async queue already handles that fine, and KEDA scale-from-zero was proven for exactly this kind of burst load in v1.6). |

    └──requires──> ffmpeg/ffprobe already present in container (shared with the normalize step above)
    └──enhances──> AUDIO_ENGINE_TIMEOUT sizing (declared-duration reject happens before the expensive

### Dependency Notes

- **ffmpeg normalize requires a new container, not a build flag:** the alternative (`WHISPER_COMMON_FFMPEG=yes` linking libavformat directly into whisper-cli) was considered and rejected as the recommendation — it's a nonstandard build not available in prebuilt whisper.cpp packages, and it collapses format-decoding and the model-inference step into one opaque binary invocation, whereas a separate `ffmpeg` → `whisper-cli` pipeline keeps each external-tool boundary explicit and independently hardened, matching how `runCommand` is already used (one call per tool, e.g. LibreOffice then a separate veraPDF check).
- **`target=json` enables SEED-001 without re-touching the Converter contract later:** this is the single highest-leverage decision in this research. If word-level timestamps and segment boundaries are included in the `json` target from day one, the future mistake-analysis milestone is purely a *consumer* of stored transcript JSON (read from S3/results, same as any other job output) — no `Converter`/`Registry`/opts change required later. Getting this wrong (e.g. shipping only `txt`/`srt` now) would force a breaking contract change exactly when SEED-001's next milestone starts.
- **AUDIO_ENGINE_TIMEOUT terminal-vs-transient is a real per-engine decision, not boilerplate:** the image engine treats timeout as transient (cheap operations, retry might just be a transient resource blip); the document engine already treats it as terminal (LibreOffice on a pathological document will time out again identically). Audio duration is user-controlled and CPU-bound-by-design (whisper.cpp real-time-factor scales roughly linearly with model size and audio length on CPU-only hardware — MEDIUM confidence, exact RTF depends on model choice and container CPU allocation and should be measured against the actual `docker-compose`/chart resource limits before picking a default timeout), so terminal (document-style) is the safer default; retries would burn a second multi-minute timeout window for a deterministically-identical outcome.
- **Diarization conflicts with the locked-engine decision:** PROJECT.md's Key Decisions table already establishes a pattern of accepting residual risk / deferring rather than reaching for a heavier dependency mid-milestone (see the PDF/A veraPDF-vs-sanity-check and chromium `file://` residual-risk entries) — treating diarization the same way (defer, don't bolt on an unreliable English-only signal) is consistent with that established decision style, not a new precedent.

## MVP Definition

### Launch With (v1.7)

- [ ] mp3/wav/m4a/ogg/opus input, normalized via ffmpeg to 16kHz mono WAV before whisper-cli — covers the realistic phone/Zoom/voice-note source set for lesson recordings
- [ ] Magic-bytes content validation for all five input containers (mp3's lack of a hard signature must be explicitly decided, not silently skipped)
- [ ] `target_format` ∈ {`txt`, `srt`, `vtt`, `json`} as ordinary `Converter.Pairs()`, no new API surface
- [ ] `json` target includes segment-level start/end/text at minimum, word-level timestamps if the pinned whisper.cpp version supports them cleanly via `-ojf` — this is the SEED-001 forward-compatibility line item and should not be cut for scope even though nothing consumes it yet
- [ ] `AudioOpts{Language, Translate}` via the existing validated-opts mechanism, default `Language="auto"` (explicitly overriding whisper.cpp's own `en` default)
- [ ] Upfront duration guard via `ffprobe` (reject absurdly long/misdeclared files before the expensive step), sized generously above 60 min for real lesson recordings
- [ ] Dedicated `audio` asynq queue, `cmd/audio-worker` binary/container, KEDA ScaledObject, reconciler routing entry — mirrors the document-engine-class precedent exactly
- [ ] `AUDIO_ENGINE_TIMEOUT` classified terminal (document precedent), not transient (image precedent)

### Add After Validation (v1.x)

- [ ] Named presets for common transcription configs (auto-detect+json, translate-to-english+srt) — trivial once `AudioOpts` exists, but not required for the first working vertical slice
- [ ] Confidence/probability fields surfaced in the `json` contract if the pinned whisper.cpp version's `-ojf` output includes them cleanly — nice-to-have enrichment of the SEED-001-facing contract, not a blocker for shipping it

### Future Consideration (v2+)

- [ ] Finer-grained streaming progress via `--print-progress` — needs a new streaming-exec primitive; defer until poll-based progress is confirmed to be a real UX complaint, not assumed
- [ ] Real speaker diarization — needs a real (non-English-only, non-stereo-dependent) diarization approach; likely its own engine-class/milestone decision, not an opt on this converter
- [ ] Arbitrary-target-language translation — needs a separate MT engine/model; whisper.cpp can only translate to English
